# Cheerful IG DM API Setup — Lessons Learned

Everything we learned setting up Instagram DM webhooks from scratch on 2026-03-01. This is the definitive guide for when we build the production webhook receiver and context engine tools.

## The Setup That Actually Worked

### 1. Meta App Creation

- Go to **developers.facebook.com/apps** (NOT developers.meta.com — the Facebook domain has the classic dashboard)
- Create a **Business** type app
- The new dashboard flow uses **"Use Cases"** with built-in steps — no separate "Add Product" sidebar anymore
- The IG messaging setup walks you through: permissions → access tokens → webhooks → business login

### 2. Instagram Account Requirements

- Must be **Business** type (not Personal, not Creator)
  - Instagram app > Settings > Account type and tools > Switch to Professional Account > Business
- Must be linked to a **Facebook Page**
  - Instagram app > Edit Profile > Public business information > Page > Connect/Create a Page
- Must be linked to a **Facebook Account**
  - Settings > Accounts Center > Add accounts
- The Facebook account, Facebook Page, and Instagram account should all be in the **same Meta Business Manager** if possible

### 3. App Roles & The "Insufficient Developer Role" Error

This was the biggest blocker. The dashboard's "Generate Token" button kept returning "Insufficient Developer Role."

**What DIDN'T work:**
- Adding Facebook account as Admin on the app alone
- Adding Instagram Tester roles (tester invite tab doesn't appear on newer IG Business accounts)
- Linking the app to Business Manager from the BM side (couldn't find the UI)
- Creating new apps (same error persisted)

**What DID work:**
The Instagram account needed the full chain connected BEFORE generating a token:
1. Instagram Business account linked to a **Facebook Account** (via Accounts Center)
2. That Facebook Account linked to a **Facebook Page** (created from IG app: Edit Profile > Page)
3. That Facebook Account is **Admin** on the Meta developer app
4. All three (IG account, FB Page, FB account) visible in the **same Business Manager**

Once this chain was complete, the dashboard's "Generate Token" button in Step 2 worked. The error is misleading — "Insufficient Developer Role" really means "the IG-to-Page-to-FB chain isn't fully connected."

### 4. Token Types — CRITICAL DISTINCTION

| Token prefix | Type | Generated by | Works with |
|---|---|---|---|
| `EAA...` | Facebook User/Page Access Token | Facebook Login OAuth | `graph.facebook.com` endpoints |
| `IGAA...` | Instagram User Access Token | Instagram Business Login | `graph.instagram.com` endpoints |

The dashboard's "Generate Token" with Instagram Business Login gives you an **`IGAA` token**. This is NOT the same as a Facebook token. Use the correct base URL:

```
# IGAA tokens → Instagram Graph API
https://graph.instagram.com/me?fields=id,username&access_token=IGAA...

# EAA tokens → Facebook Graph API
https://graph.facebook.com/me?fields=id,name&access_token=EAA...
```

Using the wrong base URL gives `Invalid OAuth 2.0 Access Token` — very misleading.

### 5. Token in .env — Hidden Character Gotcha

When loading tokens via `source .env`, hidden characters (carriage returns, etc.) can corrupt the token. The fix:

```bash
# DON'T do this — may include hidden chars
source .env
curl "...?access_token=$IG_LEAKY_ACCESS_TOKEN"

# DO this — strip hidden chars
TOKEN=$(sed -n 's/^IG_LEAKY_ACCESS_TOKEN=//p' .env | tr -d '\r\n ')
curl "...?access_token=${TOKEN}"
```

### 6. Token Lifetime

The token from the dashboard was already **long-lived (~60 days)** — no exchange needed. Token debug info showed:
- Type: User
- Expires: ~2 months out
- Valid: True
- Scopes: `instagram_business_basic`, `instagram_business_manage_messages`, `instagram_business_content_publish`, `instagram_business_manage_insights`, `instagram_business_manage_comments`

To refresh before expiry:
```
GET https://graph.instagram.com/refresh_access_token
  ?grant_type=ig_refresh_token
  &access_token={token}
```

### 7. Permissions — Instagram Business Login Flow

The new Instagram Business Login uses different permission names than the Facebook Login flow:

| Instagram Business Login | Facebook Login equivalent |
|---|---|
| `instagram_business_basic` | `instagram_basic` |
| `instagram_business_manage_messages` | `instagram_manage_messages` |
| `instagram_business_manage_comments` | `instagram_manage_comments` |
| `instagram_business_content_publish` | `instagram_content_publish` |
| `instagram_business_manage_insights` | `instagram_manage_insights` |

### 8. Webhook Setup

**Callback URL:** Any HTTPS endpoint (we used Fly.io: `https://app-name.fly.dev/webhooks/instagram`)

**Verify token:** Any arbitrary string you define — must match between dashboard and your server code.

**Webhook verification flow:**
1. Meta sends GET with `hub.mode=subscribe`, `hub.verify_token`, `hub.challenge`
2. Verify the token matches
3. Return the `hub.challenge` value as plain text with 200 status

**Subscribing to fields:** In the dashboard, check `messages`, `message_echoes`, and `messaging_postbacks`.

**Confirmed via API:**
```bash
GET https://graph.instagram.com/me/subscribed_apps?access_token={token}
# Returns: { "data": [{ "id": "...", "subscribed_fields": ["messages"] }] }
```

### 9. Dev Mode vs Live Mode — THIS WAS THE BLOCKER

**In Development Mode:** Webhooks DO NOT fire for any DMs. Not even from admin/tester accounts. We verified this — subscription was confirmed via API, webhook verification passed, but zero POST events arrived.

**In Live Mode:** Webhooks fire immediately for all inbound DMs. The moment we flipped to Live, events started flowing.

**To go Live:**
- Toggle at the top of the Meta app dashboard: Development → Live
- Requires a Privacy Policy URL (can be a simple placeholder page)
- For apps that only build for your own Instagram businesses, **App Review is NOT required** — you can skip it
- The dashboard confirms: "You can skip this step if you're a direct developer who only builds for your own Instagram businesses and don't plan to create solutions for clients."

**For Cheerful (building for clients):** Will need App Review eventually, but can develop and test in Live mode with own accounts first.

### 10. Webhook Payload — What We Actually Received

Sender IGSID: `1500760381475948` (opaque, per-app identifier)
Messages received: `"heyaaa wowzamaticlessss"`, `"gayakgayaka"`, `"wkakkakakaka"`

Key observations:
- Multiple events can arrive rapidly (sub-second apart)
- Each event is a separate POST
- IGSID is consistent per sender
- Message ID (mid) is a long base64-encoded string
- Meta retries aggressively on non-200 responses (we saw rapid-fire retries when signature verification was failing)

### 11. HMAC Signature Verification

Meta signs every POST with `X-Hub-Signature-256: sha256={hex-digest}` using HMAC-SHA256 with your **App Secret**.

**CRITICAL:** The app secret must be from the **same app** that owns the webhook. We had the secret from `cheerful-testing` but the webhook was on `cheerful-testing-2-IG` — every signature failed.

```python
import hmac, hashlib
expected = "sha256=" + hmac.new(APP_SECRET.encode(), body, hashlib.sha256).hexdigest()
hmac.compare_digest(signature, expected)
```

### 12. Rate Limits (per the spec loops)

- **200 DMs/hour per Instagram account** (outbound)
- **200 Graph API calls/hour** for IGSID resolution
- Rate limits are **per account, not per app** — each client's account has independent limits
- Inbound webhooks have no rate limit on receiving

### 13. Fly.io Deployment Notes

- `auto_stop_machines = "stop"` works but means machines spin down between webhooks — cold start is ~1-2 seconds
- For production, set `min_machines_running = 1` to keep at least one machine warm
- Meta expects webhook responses within **5-10 seconds** — cold start + processing must stay under this
- Two machines deployed by default for HA — one auto-stopped leaving one running

## Production Checklist

When building the real webhook receiver for Cheerful:

- [ ] Use the correct **App Secret** matching the production Meta app
- [ ] Enable HMAC signature verification (we disabled it for testing)
- [ ] Set `min_machines_running = 1` on Fly.io to avoid cold start timeouts
- [ ] Return 200 immediately, process async (queue or background task)
- [ ] Deduplicate by `message.mid` — Meta retries on failure, events can arrive multiple times
- [ ] Resolve IGSID to username via `GET /{igsid}?fields=name,username,profile_pic` and cache results
- [ ] Store media URLs immediately — they expire in ~1 hour
- [ ] Track 24-hour messaging window per conversation for outbound replies
- [ ] Use `graph.instagram.com` base URL with `IGAA` tokens, NOT `graph.facebook.com`
- [ ] Handle the `source .env` hidden character issue — use `sed | tr -d '\r\n'` when loading tokens
