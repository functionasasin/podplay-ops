## TypeScript Interfaces

```typescript
/**
 * An inter vivos donation made by the decedent during their lifetime.
 * Maps 1:1 to Rust `Donation` struct (types.rs:467–487).
 *
 * Pipeline consumption:
 * - step4_estate_base.rs:75–142: each donation evaluated for collatability;
 *   collatable amounts added to net_estate to produce estate_base for
 *   legitime computations. Decision tree at step4:156–301 evaluates 11
 *   exemption/special flags in priority order.
 * - step6_validation.rs:491–697: inofficiousness check — if testamentary
 *   dispositions exceed FP_disposable, Phase 3 reduction reverses donations
 *   in reverse chronological order (Art. 911 ¶1, step6:674–697).
 * - step7_distribute.rs:81–82: donations passed through for Step 8.
 * - step8_collation.rs:106–218: each donee-heir's collatable donations
 *   imputed (subtracted) from their gross entitlement to determine what
 *   they actually receive from the physical estate.
 *
 * Legal basis: Arts. 908–912 (collation/inofficiousness), Arts. 1061–1077
 * (collation rules), specific articles per exemption flag documented below.
 *
 * IMPORTANT: value_at_time_of_donation uses Money (centavos). Frontend
 * displays pesos but serializes centavos. Art. 1071 mandates valuation at
 * time of donation, NOT at time of death.
 */
export interface Donation {
  /**
   * Unique identifier for this donation.
   * Engine type: DonationId (String alias, types.rs:9).
   * Auto-generated by frontend (e.g., "don1", "don2", ...).
   */
  id: string;

  /**
   * ID of the heir who received the donation, if any.
   * Engine type: Option<HeirId> (types.rs:469).
   * null when recipient_is_stranger=true.
   * When set, must reference a valid Person.id in the family tree.
   * Engine warns "unknown_donee" if ID not found in classified heirs
   * (step4:97–113) — treated as non-collatable.
   */
  recipient_heir_id: string | null;

  /**
   * Whether the recipient is a stranger (not in the family tree).
   * Engine type: bool (types.rs:470).
   * Stranger donations are ALWAYS collatable, charged to free portion
   * (Art. 909 ¶2, step4:81–92). Not imputed against any heir's share
   * in Step 8 (step8:239–241).
   * Mutually exclusive with recipient_heir_id: if true, recipient_heir_id
   * must be null; if false, recipient_heir_id should be set.
   */
  recipient_is_stranger: boolean;

  /**
   * Value of the donation at the time it was given.
   * Engine type: Money (types.rs:471).
   * JSON: { centavos: number } — frontend displays as pesos (÷100).
   * Art. 1071: donations valued at time given, NOT at death (step8:8).
   * Must be > 0 (a zero-value donation is meaningless).
   */
  value_at_time_of_donation: Money;

  /**
   * Date the donation was made (ISO 8601 "YYYY-MM-DD").
   * Engine type: Date (String alias, types.rs:472).
   * Critical for Phase 3 inofficiousness reduction: donations reduced
   * in reverse chronological order, most recent first (step6:680).
   * Should be ≤ decedent.date_of_death.
   */
  date: string;

  /**
   * Free-text description of the donation.
   * Engine type: String (types.rs:473).
   * Informational only — not consumed by any pipeline logic.
   * Passed through to output narratives.
   */
  description: string;

  // ── Exemption/Special Flags ────────────────────────────────────
  // These 11 boolean/optional flags control the collatability decision
  // tree in step4_estate_base.rs:156–301. Priority order matters:
  // the engine checks them top-to-bottom, first match wins.

  /**
   * Donor expressly exempted this donation from collation.
   * Art. 1062 (step4:265–268).
   * NOT collatable for estate base, BUT still checked for
   * inofficiousness (still_check_inofficiousness=true).
   * Default: false.
   */
  is_expressly_exempt: boolean;

  /**
   * Donation was for support, education, or medical expenses.
   * Art. 1067 (step4:196–198).
   * NOT collatable — fully exempt.
   * Default: false.
   */
  is_support_education_medical: boolean;

  /**
   * Donation was a customary/ordinary gift.
   * Art. 1067 (step4:196–198).
   * NOT collatable — fully exempt.
   * Default: false.
   */
  is_customary_gift: boolean;

  /**
   * Donation was for professional or vocational education.
   * Art. 1068 (step4:234–257).
   * Conditionally collatable: exempt by default, BUT collatable if
   * professional_expense_parent_required=true. When collatable,
   * imputed savings deducted: collatable = value - imputed_savings.
   * Default: false.
   */
  is_professional_expense: boolean;

  /**
   * Whether the parent was legally required to provide this
   * professional education. Only relevant when is_professional_expense=true.
   * Art. 1068 (step4:235–237).
   * false → donation exempt (Art. 1068 default).
   * true → donation collatable (minus imputed savings).
   * Default: false.
   */
  professional_expense_parent_required: boolean;

  /**
   * Imputed savings the child would have spent living at home.
   * Only relevant when is_professional_expense=true AND
   * professional_expense_parent_required=true.
   * Engine type: Option<Money> (types.rs:479).
   * JSON: null or { centavos: number }.
   * Collatable amount = value - imputed_savings (step4:239–247).
   * If savings ≥ donation value, nothing collated.
   * Default: null.
   */
  professional_expense_imputed_savings: Money | null;

  /**
   * Donation was given jointly by both parents (not just the decedent).
   * Art. 1072 (step4:220–231).
   * HALF collatable: only ½ charged to this estate (other half
   * belongs to the other parent's estate).
   * Default: false.
   */
  is_joint_from_both_parents: boolean;

  /**
   * Donation was made to the child's spouse only (not the child).
   * Art. 1066 (step4:201–203).
   * NOT collatable — fully exempt.
   * Default: false.
   */
  is_to_child_spouse_only: boolean;

  /**
   * Donation was a joint gift to both the child and their spouse.
   * Art. 1066 ¶2 (step4:206–217).
   * HALF collatable: only the child's half is collated.
   * Default: false.
   */
  is_joint_to_child_and_spouse: boolean;

  /**
   * Donation was a wedding gift.
   * Art. 1070 (step4:261–263).
   * NOT collatable for estate base computation, BUT deferred
   * inofficiousness check (still_check_inofficiousness=true).
   * Default: false.
   */
  is_wedding_gift: boolean;

  /**
   * Donation was payment of a debt owed by the child.
   * Art. 1069 (step4:276–289).
   * ALWAYS collatable — cannot be exempted.
   * Default: false.
   */
  is_debt_payment_for_child: boolean;

  /**
   * Donation was for election campaign expenses.
   * Art. 1069 (step4:276–289).
   * ALWAYS collatable — cannot be exempted.
   * Default: false.
   */
  is_election_expense: boolean;

  /**
   * Donation was payment of a fine imposed on the child.
   * Art. 1069 (step4:276–289).
   * ALWAYS collatable — cannot be exempted.
   * Default: false.
   */
  is_fine_payment: boolean;
}
```

## Zod Validation Schema

```typescript
import { z } from "zod";
import { MoneySchema } from "./money";

/**
 * Zod schema for Donation.
 * Constraints sourced from:
 * - types.rs:467–487 (struct fields)
 * - step4_estate_base.rs:156–301 (collatability decision tree)
 * - step6_validation.rs:674–697 (date ordering for reduction)
 * - step4:97–113 (unknown_donee warning)
 */
export const DonationSchema = z.object({
  /** Auto-generated, non-empty (types.rs:468). */
  id: z.string().min(1, "Donation ID is required"),

  /**
   * Must reference a valid Person.id when recipient_is_stranger=false.
   * Engine warns if not found (step4:97–113).
   */
  recipient_heir_id: z.string().nullable(),

  /** Mutually exclusive with recipient_heir_id (step4:81–126). */
  recipient_is_stranger: z.boolean().default(false),

  /**
   * Must be > 0 centavos. Art. 1071: valued at time of donation.
   * Frontend collects pesos, converts to centavos for serialization.
   */
  value_at_time_of_donation: MoneySchema.refine(
    (m) => m.centavos > 0,
    "Donation value must be greater than zero"
  ),

  /**
   * ISO 8601 date. Used for Phase 3 reduction ordering (step6:680).
   * Cross-validated: must be ≤ decedent.date_of_death.
   */
  date: z.string().regex(
    /^\d{4}-\d{2}-\d{2}$/,
    "Date must be in YYYY-MM-DD format"
  ),

  /** Free-text, informational (types.rs:473). */
  description: z.string().default(""),

  // ── Exemption flags (all default false) ──
  is_expressly_exempt: z.boolean().default(false),
  is_support_education_medical: z.boolean().default(false),
  is_customary_gift: z.boolean().default(false),
  is_professional_expense: z.boolean().default(false),
  professional_expense_parent_required: z.boolean().default(false),
  professional_expense_imputed_savings: MoneySchema.nullable().default(null),
  is_joint_from_both_parents: z.boolean().default(false),
  is_to_child_spouse_only: z.boolean().default(false),
  is_joint_to_child_and_spouse: z.boolean().default(false),
  is_wedding_gift: z.boolean().default(false),
  is_debt_payment_for_child: z.boolean().default(false),
  is_election_expense: z.boolean().default(false),
  is_fine_payment: z.boolean().default(false),
}).superRefine((data, ctx) => {
  // Cross-field: recipient_is_stranger ↔ recipient_heir_id mutual exclusion
  // (step4:81–126 — three-way branch: stranger → heir_id → neither)
  if (data.recipient_is_stranger && data.recipient_heir_id !== null) {
    ctx.addIssue({
      code: z.ZodIssueCode.custom,
      message: "Stranger donation cannot have a recipient heir ID",
      path: ["recipient_heir_id"],
    });
  }
  if (!data.recipient_is_stranger && data.recipient_heir_id === null) {
    ctx.addIssue({
      code: z.ZodIssueCode.custom,
      message: "Non-stranger donation must specify a recipient heir",
      path: ["recipient_heir_id"],
    });
  }

  // Cross-field: professional_expense sub-fields only relevant when
  // is_professional_expense=true (step4:234–257)
  if (!data.is_professional_expense) {
    if (data.professional_expense_parent_required) {
      ctx.addIssue({
        code: z.ZodIssueCode.custom,
        message: "Parent-required flag only applies to professional expenses",
        path: ["professional_expense_parent_required"],
      });
    }
    if (data.professional_expense_imputed_savings !== null) {
      ctx.addIssue({
        code: z.ZodIssueCode.custom,
        message: "Imputed savings only applies to professional expenses",
        path: ["professional_expense_imputed_savings"],
      });
    }
  }

  // Cross-field: imputed_savings only relevant when parent_required=true
  // (step4:239–247)
  if (data.is_professional_expense &&
      !data.professional_expense_parent_required &&
      data.professional_expense_imputed_savings !== null) {
    ctx.addIssue({
      code: z.ZodIssueCode.custom,
      message: "Imputed savings only applies when parent was required to provide education",
      path: ["professional_expense_imputed_savings"],
    });
  }

  // Mutual exclusion: only one special category flag should be true at a time.
  // The engine's decision tree (step4:156–301) checks top-to-bottom and takes
  // the first match, but the frontend should prevent ambiguous combinations.
  const specialFlags = [
    data.is_support_education_medical,
    data.is_customary_gift,
    data.is_professional_expense,
    data.is_to_child_spouse_only,
    data.is_joint_to_child_and_spouse,
    data.is_joint_from_both_parents,
    data.is_wedding_gift,
    data.is_debt_payment_for_child,
    data.is_election_expense,
    data.is_fine_payment,
  ].filter(Boolean).length;

  if (specialFlags > 1) {
    ctx.addIssue({
      code: z.ZodIssueCode.custom,
      message: "Only one donation category flag should be active at a time",
      path: ["is_support_education_medical"], // highlight first flag
    });
  }
});
```

## Conditional Visibility Rules

### Top-level: Donations step

The Donations step is always available (both testate and intestate estates can have inter vivos donations). The `EngineInput.donations` field is a `Vec<Donation>` (can be `[]`).

### Recipient mode toggle

```
recipient_is_stranger = false  →  Show "Recipient" person picker (select from family tree)
recipient_is_stranger = true   →  Hide person picker, set recipient_heir_id = null
```

### Exemption flags (only for non-stranger donations)

All 11 exemption/special-category flags are **hidden** when `recipient_is_stranger = true`. Stranger donations are always collatable (Art. 909 ¶2) — exemption flags are irrelevant.

### Professional expense cascade

```
is_professional_expense = false  →  Hide professional_expense_parent_required
                                    Hide professional_expense_imputed_savings

is_professional_expense = true   →  Show professional_expense_parent_required
    professional_expense_parent_required = false  →  Hide imputed_savings
    professional_expense_parent_required = true   →  Show imputed_savings (MoneyInput)
```

### Mutual exclusion of donation categories

The engine's decision tree (step4:156–301) processes flags top-to-bottom, first-match-wins. Allowing multiple flags would produce technically valid but misleading results. The frontend should enforce radio-group-like behavior:

**Category groups** (at most one active):

| Category | Legal Basis | Collatability |
|----------|-------------|---------------|
| _(none selected — standard donation)_ | Art. 909/910 | Fully collatable |
| Support/Education/Medical | Art. 1067 | Exempt |
| Customary Gift | Art. 1067 | Exempt |
| Professional/Vocational Education | Art. 1068 | Conditional (see cascade) |
| Child's Spouse Only | Art. 1066 | Exempt |
| Joint Gift to Child + Spouse | Art. 1066 ¶2 | Half collatable |
| Joint from Both Parents | Art. 1072 | Half collatable |
| Wedding Gift | Art. 1070 | Exempt (but inofficiousness checked) |
| Payment of Child's Debt | Art. 1069 | Always collatable |
| Election Campaign Expense | Art. 1069 | Always collatable |
| Payment of Child's Fine | Art. 1069 | Always collatable |

**Recommended UI**: A select/dropdown for "Donation Category" with an "(None — Standard Advance)" default, mapping to the 11 boolean flags. Only `is_expressly_exempt` is orthogonal (can be combined with a category — it's a separate donor instruction, not a category).

## Collatability Decision Tree (Engine Reference)

The frontend should display a summary of the collatability outcome. The engine's logic (step4:156–301) follows this priority order:

```
1. recipient_is_stranger?          → COLLATABLE, charged to Free Portion (Art. 909 ¶2)
2. recipient not found in heirs?   → NOT COLLATABLE, warning "unknown_donee"
3. heir not compulsory?            → NOT COLLATABLE (Art. 1061)
4. sole compulsory heir?           → NOT COLLATABLE (Art. 1061)
5. is_support_education_medical    → NOT COLLATABLE (Art. 1067)
   OR is_customary_gift?
6. is_to_child_spouse_only?        → NOT COLLATABLE (Art. 1066)
7. is_joint_to_child_and_spouse?   → HALF COLLATABLE (Art. 1066 ¶2)
8. is_joint_from_both_parents?     → HALF COLLATABLE (Art. 1072)
9. is_professional_expense?
   a. parent NOT required           → NOT COLLATABLE (Art. 1068 default)
   b. parent required               → COLLATABLE (value - imputed_savings) (Art. 1068)
10. is_wedding_gift?               → NOT COLLATABLE (Art. 1070), still check inofficiousness
11. is_expressly_exempt?           → NOT COLLATABLE (Art. 1062), still check inofficiousness
12. heir has_renounced?            → NOT COLLATABLE (Art. 1062 ¶2), still check inofficiousness
13. is_debt_payment_for_child      → ALWAYS COLLATABLE (Art. 1069)
    OR is_election_expense
    OR is_fine_payment?
14. DEFAULT (standard)             → FULLY COLLATABLE (Art. 909/910)
```

## Serialization Notes

### Money as centavos

`value_at_time_of_donation` and `professional_expense_imputed_savings` serialize as `{ "centavos": <number> }`. Frontend collects pesos, multiplies by 100 for serialization.

```typescript
// Frontend → Engine
{ centavos: pesoAmount * 100 }

// Engine → Frontend (display)
₱ (centavos / 100).toLocaleString("en-PH", { minimumFractionDigits: 2 })
```

### Date as ISO string

The `date` field is a plain `"YYYY-MM-DD"` string. No timezone handling needed.

### Null vs absent

`recipient_heir_id` serializes as JSON `null` (not omitted) when the recipient is a stranger. `professional_expense_imputed_savings` serializes as JSON `null` when not applicable.

### Boolean flags

All 11 boolean flags serialize as JSON `true`/`false`. All default to `false`. No special serde tagging.

## Edge Cases

1. **Zero donations**: `EngineInput.donations` can be `[]` (and is `[]` in most test cases). The Donations wizard step should show an empty repeater with an "Add Donation" button.

2. **Stranger donation with flags set**: Engine ignores exemption flags for stranger donations (step4:81–92 takes the stranger branch before checking any flags). Frontend should hide/disable flags when `recipient_is_stranger=true`.

3. **Unknown donee**: If `recipient_heir_id` references a person not in `family_tree`, engine generates a warning but treats donation as non-collatable (step4:97–113). Frontend should cross-validate against current family tree persons.

4. **Sole compulsory heir exemption**: Even with flags, a donation to the sole compulsory heir is never collatable (step4:186–193, Art. 1061). This is not checkable at donation entry time since it depends on the full family tree classification.

5. **Expressly exempt + inofficiousness**: `is_expressly_exempt` prevents collation (not added to estate base), but engine still flags for inofficiousness check (step4:265–268, `still_check_inofficiousness=true`). Same for `is_wedding_gift` (step4:261–263) and renounced heirs (step4:270–273).

6. **Professional expense edge case — savings ≥ value**: When `professional_expense_imputed_savings ≥ value_at_time_of_donation`, collatable amount is 0 and donation treated as non-collatable (step4:245–247).

7. **Date ordering for inofficiousness reduction**: Phase 3 of inofficiousness resolution reduces donations in reverse chronological order by `date` (step6:680 — `b.date.cmp(&a.date)`). This is string comparison on ISO dates, which works correctly for YYYY-MM-DD format.

8. **Multiple donations to same heir**: Allowed and common in test cases (fuzz-cases/076 has 3 donations to 3 different heirs). Step 8 sums all collatable amounts per heir (step8:226–243).

9. **Donation to dead heir**: If the recipient heir is deceased and has representatives, the parent's donations are collated against the representation line's entitlement (Art. 1064, step8:128–186). Each representative bears a proportional share.

10. **Half-collatable edge cases**: `is_joint_to_child_and_spouse` and `is_joint_from_both_parents` both result in exactly ½ of the donation value being collatable (step4:207 and step4:221). These are independent — it's theoretically possible for a donation to be both joint-from-parents and joint-to-child-and-spouse, but the engine's first-match logic would apply only `is_joint_to_child_and_spouse` (checked first at step4:206).

## Wizard Step Placement

The Donations step appears as a **standalone step** in the wizard, available for both testate and intestate estates. It consists of a repeater (dynamic list) where each item is a single Donation form.

**Ordering within the repeater item:**
1. Recipient mode: stranger toggle + person picker
2. Core fields: value, date, description
3. Category selector (maps to the 11 boolean flags)
4. Express exemption toggle (orthogonal to category)
5. Professional expense sub-fields (conditional)

**Dependencies**: Family tree step must be completed before Donations (for person picker options and recipient validation).
