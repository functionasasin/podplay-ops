## 2. Detailed Risk Analysis Per Plugin

### 2.1 `vite-plugin-top-level-await` — HIGH RISK

**What it does:** Transforms top-level `await` expressions in ES modules into wrapper code compatible with older browsers/bundlers. In practice, it wraps each module that contains top-level await in an immediately-invoked async function.

**What can go wrong:** The wrapping can disrupt JavaScript's prototype chain for objects whose `constructor` or `prototype` assignment happens at module initialization time. The `wasm-bindgen`-generated `taxklaro_engine.js` glue code uses top-level await internally for `init()`. If the transformation corrupts the prototype of a WASM-exported class, all calls to that class fail silently in production.

**Known affected libraries:**
- `d3-*`: Uses prototype chain for scale objects. If TaxKlaro uses d3 for charts, test explicitly.
- `recharts`: Uses class inheritance for chart components. Test explicitly if added.
- `@react-pdf/renderer`: Lazy-loaded, so the top-level-await transformation may not affect it. Verify.
- The WASM glue code itself: `taxklaro_engine.js` (generated by wasm-pack). This is the primary concern.

**Mitigation — already specified:** Set `build.target = "es2022"` in `vite.config.ts`. This tells `vite-plugin-top-level-await` that the output environment supports native top-level await, so the plugin does minimal transformation (or none at all for fully es2022-capable targets). The risk is lower with `es2022` target but NOT eliminated — the plugin still transforms for older chunks.

**Mandatory verification:** Run the production smoke test (Section 4). Specifically, the "Compute Tax" step must succeed. If it fails, check browser console for `TypeError: ... is not a constructor` or `prototype` errors.

**Workaround if prototype errors occur:** Pin the affected library to a version that doesn't trigger the issue, or exclude it from the plugin's transformation scope:

```typescript
// vite.config.ts workaround (only if prototype errors are confirmed in smoke test)
topLevelAwait({
  // Exclude known-good modules from transformation
  // Only add exclusions if smoke test confirms a specific module is broken
}),
```

### 2.2 `vite-plugin-wasm` — MEDIUM RISK

**What it does:** Intercepts `.wasm` file imports, emits the binary as a separate asset in `dist/assets/`, and rewrites the `init()` call to fetch the asset URL at runtime.

**What can go wrong:**

1. **WASM URL mismatch in production:** In dev mode, the plugin serves `.wasm` from memory. In production, it emits to `dist/assets/taxklaro_engine_bg-[hash].wasm`. The `init()` call in the generated glue code must use the correct URL. If the URL is wrong, the browser fetches 404 and `computeTax()` fails with a network error.

   **Verification:** After `npm run build`, check that `dist/assets/` contains a `.wasm` file. Open `dist/assets/taxklaro_engine*.js` and verify the `import.meta.url`-based fetch URL is present.

2. **`--target web` vs `--target bundler`:** The wasm-pack target MUST be `web`, not `bundler`. With `--target bundler`, the generated glue uses `new URL(..., import.meta.url)` in a way that `vite-plugin-wasm` does not intercept correctly, causing the WASM file to be missing from the build output.

   **Verification:** In `package.json` scripts, confirm `wasm-pack build --target web`. Never `--target bundler`.

3. **WASM binary not found in dist:** If the engine is not built before `npm run build`, the `src/wasm/pkg/` directory is empty or stale. The build will succeed (no import error at build time) but the runtime fetch will 404.

   **Mitigation:** The `build` script MUST run `wasm-pack build` before `vite build`:

   ```json
   {
     "scripts": {
       "build:wasm": "cd ../engine && wasm-pack build --target web --out-dir ../frontend/src/wasm/pkg",
       "build": "npm run build:wasm && vite build",
       "dev": "npm run build:wasm && vite",
       "preview": "vite preview"
     }
   }
   ```

### 2.3 `@tailwindcss/vite` — MEDIUM RISK

**What it does:** Scans source files for Tailwind utility class names and emits a CSS file containing only the used classes.

**What can go wrong:** If the plugin's file scanner doesn't cover all source file locations, it may emit a nearly-empty CSS file. All Tailwind classes in those files will be absent from production, causing the app to render with zero styling.

**Verification:** After `npm run build`, run:
```bash
ls -lh dist/assets/*.css
```
The CSS file must be **> 20KB** (typically 40–100KB for a real app). If it's < 5KB, the scanner missed most files.

**Default scanner config (usually works):** `@tailwindcss/vite` auto-discovers content from `./src/**/*.{ts,tsx}`. No additional configuration required if all components are in `src/`.

**If scanner misses files:** Add explicit content paths in `tailwind.config.ts`:
```typescript
// tailwind.config.ts (only if auto-discovery fails)
export default {
  content: [
    "./src/**/*.{ts,tsx}",
    "./index.html",
  ],
};
```

## 4. Mandatory Production Build Smoke Test (Phase 7)

This is the **required** production verification sequence. The forward loop's Phase 7 (Completeness Verification) MUST execute ALL of these steps in order. A failure at any step is a blocker.

### Step 1: Clean Build

```bash
# From the frontend/ directory
rm -rf dist/
npm run build
# Expected: exits 0, no errors, no warnings about missing modules
```

**What can go wrong:**
- `[vite-plugin-wasm] Cannot find module './pkg/taxklaro_engine.js'` → WASM not built. Run `npm run build:wasm` first.
- `error TS2345: Argument of type 'X' is not assignable to type 'Y'` → TypeScript type error in source.
- `Rollup failed to resolve import` → Missing dependency in `package.json`.

### Step 2: Verify Build Output

```bash
# Check for required output files
ls dist/
# Expected: index.html, assets/

ls dist/assets/ | grep -E '\.(js|css|wasm)$'
# Expected: at least one .js file, one .css file, one .wasm file

# Verify CSS is not empty
wc -c dist/assets/*.css
# Expected: > 20000 bytes (20KB)

# Verify WASM binary is present
ls -lh dist/assets/*.wasm
# Expected: one .wasm file, size > 100KB
```

### Step 3: Start Production Server

```bash
# Use 'serve' package (installed globally or via npx)
npx serve dist -l 4173
# Leave running in background for subsequent steps
```

### Step 4: Basic HTTP Check

```bash
# Must return HTML with <div id="root">
curl -s http://localhost:4173 | grep -c '<div id="root">'
# Expected: 1 (found exactly once)
```

### Step 5: Playwright Production Smoke Test

Run a focused Playwright test against `http://localhost:4173` (not the Vite dev server). This test MUST:

1. Open the landing page — verify it renders (no white page, no console errors)
2. Navigate to `/auth` — verify Sign In and Create Account tabs render
3. Complete the full computation wizard (using test credentials from environment)
4. Click "Compute Tax" — verify results page renders with three-regime comparison table
5. Verify the recommended regime highlight is visible
6. Click "Export PDF" — verify a PDF download is initiated (Playwright file download assertion)

**Playwright config for production smoke test:**

```typescript
// playwright.prod.config.ts
import { defineConfig } from "@playwright/test";

export default defineConfig({
  testDir: "./e2e",
  testMatch: "*.prod.spec.ts",  // Only production smoke tests
  use: {
    baseURL: "http://localhost:4173",
    screenshot: "only-on-failure",
    video: "on-first-retry",
  },
  retries: 0,  // No retries — production smoke test must pass first time
  reporter: [["list"], ["html", { outputFolder: "playwright-report-prod" }]],
```

**Production smoke test file:**

```typescript
// e2e/smoke.prod.spec.ts
import { test, expect } from "@playwright/test";

const TEST_EMAIL = process.env.E2E_TEST_EMAIL ?? "smoketest@taxklaro.ph";
const TEST_PASSWORD = process.env.E2E_TEST_PASSWORD ?? "SmokeTest123!";

test.describe("Production Build Smoke Test", () => {
  test("landing page renders without white page", async ({ page }) => {
    const consoleErrors: string[] = [];
    page.on("console", (msg) => {
      if (msg.type() === "error") consoleErrors.push(msg.text());
    });

    await page.goto("/");

    // Must not be a white page
    const rootEl = page.locator("#root");
    await expect(rootEl).not.toBeEmpty();

    // No prototype chain errors (the main failure mode of vite-plugin-top-level-await)
    const fatalErrors = consoleErrors.filter((e) =>
      e.includes("is not a constructor") ||
      e.includes("prototype") ||
      e.includes("Cannot read properties of undefined")
    );
    expect(fatalErrors).toHaveLength(0);
  });

  test("auth page renders", async ({ page }) => {
    await page.goto("/auth");
    await expect(page.getByRole("tab", { name: "Sign In" })).toBeVisible();
    await expect(page.getByRole("tab", { name: "Create Account" })).toBeVisible();
  });

  test("WASM computation works in production build", async ({ page }) => {
    // Sign in with test credentials
    await page.goto("/auth");
    await page.getByLabel("Email").fill(TEST_EMAIL);
    await page.getByLabel("Password").fill(TEST_PASSWORD);
    await page.getByRole("button", { name: "Sign In" }).click();

    // Wait for redirect to dashboard or computations
    await expect(page).toHaveURL(/\/(computations)?$/);

    // Navigate to new computation
    await page.getByRole("link", { name: "New Computation" }).click();
    await expect(page).toHaveURL("/computations/new");

    // Fill wizard — Step 1: Taxpayer Info
    await page.getByLabel("Taxpayer Name").fill("Juan dela Cruz");
    await page.getByLabel("TIN").fill("123-456-789-000");
    await page.getByLabel("Tax Year").fill("2025");
    // Select PURELY_SE
    await page.getByRole("combobox", { name: "Taxpayer Type" }).click();
    await page.getByRole("option", { name: "Purely Self-Employed" }).click();
    await page.getByRole("button", { name: "Next" }).click();

    // Fill wizard — Step 3: Gross Receipts
    await page.getByLabel("Gross Receipts").fill("1200000");
    await page.getByRole("button", { name: "Next" }).click();

    // Advance through remaining wizard steps with defaults
    // (Steps 4-8 have sensible defaults for a purely SE filer with no expenses)
    for (let i = 0; i < 5; i++) {
      const nextBtn = page.getByRole("button", { name: "Next" });
      if (await nextBtn.isVisible()) await nextBtn.click();
    }

    // Click Compute
    await page.getByRole("button", { name: "Compute Tax" }).click();

    // Wait for results — this exercises the WASM engine in production mode
    await expect(page.getByTestId("results-view")).toBeVisible({ timeout: 10000 });

    // Verify three-regime comparison is visible
    await expect(page.getByTestId("regime-comparison-table")).toBeVisible();

    // Verify recommended regime is highlighted
    await expect(page.getByTestId("recommended-regime-badge")).toBeVisible();

    // Verify no "An unexpected error occurred" message (WASM internal error)
    await expect(page.getByText("An unexpected error occurred")).not.toBeVisible();
  });

  test("CSS is not empty (Tailwind scanner worked)", async ({ page }) => {
    await page.goto("/auth");

    // The auth page must have visible styling — if Tailwind missed all files,
    // everything would be unstyled raw HTML
    const signInTab = page.getByRole("tab", { name: "Sign In" });
    await expect(signInTab).toBeVisible();

    // Check computed background color of the page body is NOT browser default (white with no style)
    const bodyBg = await page.evaluate(() =>
      window.getComputedStyle(document.body).backgroundColor
    );
    // TaxKlaro uses bg-slate-50 or bg-background — not "rgba(0, 0, 0, 0)"
    expect(bodyBg).not.toBe("rgba(0, 0, 0, 0)");
  });
});
```

## 5. Tree-Shaking-Sensitive Libraries

These libraries have known issues with tree-shaking or production bundling:

### 5.1 `@react-pdf/renderer` (LAZY-LOADED — MITIGATED)

**Risk:** `@react-pdf/renderer` uses CommonJS internally and has side effects in module initialization. Tree-shaking can strip initialization code that sets up PDF rendering context.

**Mitigation already in spec (action-trigger-map):** This library is **lazy-loaded** via `React.lazy()` + `import()` inside the `handleExportPdf` handler. It is NOT in the main bundle. The lazy load happens only when the user clicks "Export PDF."

**Production verification:** Click "Export PDF" in the smoke test. If the lazy import fails, the browser will show an error in the network tab (`/assets/pdf-[hash].js` returns 404 or throws on eval).

**Specific test step:** The `smoke.prod.spec.ts` test above does not test PDF export (requires a completed computation). Add a separate manual verification step:
1. Complete a computation in the production build
2. Click "Export PDF"
3. Verify PDF downloads and contains data (not blank)

### 5.2 `lucide-react` (SAFE — TREE-SHAKES WELL)

lucide-react exports each icon as an individual named export. Vite's tree-shaker correctly eliminates unused icons. No production risk.

**Verification:** Check `dist/assets/*.js` bundle size. Should be < 2MB gzipped. If > 5MB, tree-shaking may have failed for lucide.

### 5.3 `@tanstack/react-router` (SAFE — ESM NATIVE)

TanStack Router is ESM-native and tree-shakes correctly. File-based route code-splitting is handled automatically. No production risk beyond standard Vite bundling.

### 5.4 `zod` (SAFE)

Zod is ESM-native. Named imports tree-shake correctly.

### 5.5 `@supabase/supabase-js` (MEDIUM — LARGE BUNDLE)

The Supabase client is ~300KB minified. It does not tree-shake aggressively because of its plugin architecture. This is expected behavior, not a bug. The production bundle will include the full client.

**Verification:** `dist/assets/*.js` will include a large Supabase chunk. This is correct.

## 6. CI/CD Production Build Step

The CI pipeline (Wave 6, `ci-cd-pipeline` aspect) MUST include a production build step BEFORE deployment:

```yaml
# .github/workflows/deploy.yml (relevant steps)
- name: Build WASM engine
  run: cd engine && wasm-pack build --target web --out-dir ../frontend/src/wasm/pkg

- name: Build frontend (production)
  working-directory: frontend
  run: npm run build
  env:
    VITE_SUPABASE_URL: ${{ secrets.VITE_SUPABASE_URL }}
    VITE_SUPABASE_ANON_KEY: ${{ secrets.VITE_SUPABASE_ANON_KEY }}
    VITE_APP_URL: ${{ secrets.VITE_APP_URL }}

- name: Verify CSS output
  working-directory: frontend
  run: |
    CSS_SIZE=$(wc -c < dist/assets/*.css)
    echo "CSS bundle size: ${CSS_SIZE} bytes"
    if [ "${CSS_SIZE}" -lt 20000 ]; then
      echo "ERROR: CSS bundle is too small (${CSS_SIZE} bytes). Tailwind scanner may have failed."
      exit 1
    fi

- name: Verify WASM output
  working-directory: frontend
  run: |
    if ! ls dist/assets/*.wasm 1> /dev/null 2>&1; then
      echo "ERROR: No .wasm file in dist/assets/. WASM build may have failed."
      exit 1
    fi
    WASM_SIZE=$(wc -c < dist/assets/*.wasm)
    echo "WASM binary size: ${WASM_SIZE} bytes"
    if [ "${WASM_SIZE}" -lt 100000 ]; then
      echo "ERROR: WASM binary is too small (${WASM_SIZE} bytes). Engine may not have been compiled."
      exit 1
    fi
```

## 7. Production Failure Mode Reference

| Failure Mode | Symptom | Root Cause | Fix |
|---|---|---|---|
| White page on load | Browser shows blank. Console: `TypeError: X is not a constructor` | `vite-plugin-top-level-await` broke prototype chain | Ensure `build.target = "es2022"`. Check which library is affected. May need to pin or exclude. |
| WASM 404 in production | `computeTax()` fails. Network tab shows `taxklaro_engine_bg-*.wasm` returning 404 | `--target bundler` used instead of `--target web`, or WASM not copied to `dist/assets/` | Use `--target web`. Verify `vite-plugin-wasm` is in plugins array. |
| WASM not initialized | `computeTax()` throws before user clicks Compute | Bridge `ensureWasmInitialized()` failed silently | Check browser console for WASM fetch error. Verify WASM binary is in `dist/assets/`. |
| No CSS styling | App renders as unstyled HTML | `@tailwindcss/vite` scanner missed source files | Add explicit `content` paths to Tailwind config. |
| PDF export blank | PDF downloads but has no content or error on open | `@react-pdf/renderer` lazy-load failed or tree-shaking stripped init code | Verify lazy import syntax. Check network tab for `pdf-*.js` chunk loading. |
| Missing environment variables | Supabase calls fail with "Missing API key" | `VITE_*` vars not injected during Docker build | Pass as `--build-arg` in Dockerfile. Verify `docker build --build-arg VITE_SUPABASE_URL=...`. |
| Router shows 404 on direct URL access | Navigating directly to `/computations` returns 404 | Production server not configured for SPA fallback | Ensure `serve` is started with `-s` flag (`serve -s dist`). Or configure nginx with `try_files`. |

## 8. Serve Command: SPA Mode

**Critical:** The production server must be started with SPA mode enabled. Without it, direct URL access (e.g., bookmarking `/computations`) returns a 404 from the file server.

```bash
# CORRECT — SPA mode, all paths return index.html
npx serve dist -s -l 4173

# WRONG — without -s flag, /computations returns 404
npx serve dist -l 4173
```

In the Dockerfile, this is handled by the CMD:
```dockerfile
CMD ["serve", "-s", "dist", "-l", "8080"]
```

The `-s` flag (single-page app mode) is **mandatory**.
