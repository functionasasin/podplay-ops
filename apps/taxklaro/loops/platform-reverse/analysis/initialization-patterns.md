## 1. WASM Package Structure

After running `wasm-pack build`, the following files are generated:

```
frontend/src/wasm/pkg/
  taxklaro_engine.js          — wasm-bindgen JS glue code (imports .wasm, exports functions)
  taxklaro_engine_bg.wasm     — compiled WASM binary (~500KB–2MB depending on engine size)
  taxklaro_engine_bg.js       — internal: instantiate_wasm helper
  taxklaro_engine.d.ts        — TypeScript declarations (auto-generated)
  package.json                — marks pkg/ as an ES module package
```

**Critical detail:** `taxklaro_engine.d.ts` is auto-generated by wasm-bindgen. It declares:
- `export function compute_json(input: string): string` — the primary export
- `export function validate_json(input: string): string` — the validation export
- `export default function init(module_or_path?: ...): Promise<InitOutput>` — async browser init
- `export function initSync(module_or_path: ...): InitOutput` — sync Node.js init

The `default` export (`init`) is for browser use. `initSync` is the named export for Node.js.

## 2. wasm-pack Build Command

```bash
# From the engine/ directory (not frontend/)
cd engine
wasm-pack build --target web --out-dir ../frontend/src/wasm/pkg
```

**Target: `web` (NOT `bundler`, NOT `nodejs`)**

Rationale:
- `--target web` produces ES module output with explicit `import.meta.url`-based WASM URL resolution. This is the ONLY target compatible with `vite-plugin-wasm`.
- `--target bundler` produces code that uses `import` for the `.wasm` file, which confuses Vite's chunking and breaks in production mode.
- `--target nodejs` uses `require()` and CommonJS — incompatible with Vite's ES module pipeline.

**Build script in root `Makefile` or `package.json`:**

```json
// engine/package.json (or root Makefile)
{
  "scripts": {
    "build:wasm": "cd engine && wasm-pack build --target web --out-dir ../frontend/src/wasm/pkg",
    "dev": "npm run build:wasm && cd frontend && npm run dev",
    "build": "npm run build:wasm && cd frontend && npm run build"
  }
}
```

## 3. Vite Configuration

### `frontend/vite.config.ts`

```typescript
import { defineConfig } from "vite";
import react from "@vitejs/plugin-react";
import wasm from "vite-plugin-wasm";
import topLevelAwait from "vite-plugin-top-level-await";
import tailwindcss from "@tailwindcss/vite";
import path from "path";

export default defineConfig({
  plugins: [
    react(),
    wasm(),
    topLevelAwait(),  // Required: WASM init uses top-level await internally
    tailwindcss(),
  ],
  resolve: {
    alias: {
      "@": path.resolve(__dirname, "./src"),
    },
  },
  build: {
    target: "es2022",  // Required for top-level await support in output
  },
  test: {
    // Vitest config (inline)
    environment: "jsdom",
    globals: true,
    setupFiles: ["./src/test-setup.ts"],
  },
});
```

**Plugin notes:**
- `vite-plugin-wasm`: Intercepts `.wasm` imports, emits the binary as a separate asset, and rewrites the init call to fetch it. Without this, Vite does not know how to handle `.wasm` files.
- `vite-plugin-top-level-await`: Transforms top-level `await` expressions for browser targets that don't natively support them. The `wasm-bindgen`-generated glue code MAY use top-level await internally; this plugin ensures compatibility.
- `build.target = "es2022"`: Sets the minimum ECMAScript target for the output bundle. `es2022` includes native top-level await support, which reduces the transformation overhead of `vite-plugin-top-level-await`.

**Production build risk:** `vite-plugin-top-level-await` can break prototype chains for libraries that use `new` constructor calls in their initialization (d3, recharts). See Wave 6 `production-build-verification` for the mitigation smoke test.

## 4. The Complete `bridge.ts` File

**Location:** `frontend/src/wasm/bridge.ts`

```typescript
/**
 * TaxKlaro WASM Bridge
 *
 * This module is the ONLY place that imports from the WASM package.
 * All other code must go through this module's exported functions.
 *
 * Initialization strategy:
 * - Browser: async init(), fetches .wasm file via HTTP
 * - Node.js (vitest): initSync(), reads .wasm file from disk synchronously
 *
 * The bridge initializes once and caches the result. Subsequent calls
 * to computeTax() or validateInput() are synchronous after initialization.
 */

import type { TaxpayerInput } from "@/types/engine-input";
import type {
  TaxComputationResult,
  ValidationOnlyResult,
} from "@/types/engine-output";

// ── Initialization State ──────────────────────────────────────────────────────

let wasmInitPromise: Promise<void> | null = null;

/**
 * Ensures WASM is initialized exactly once.
 * Returns a promise that resolves when WASM is ready.
 * Safe to call concurrently — multiple callers share the same promise.
 */
async function ensureWasmInitialized(): Promise<void> {
  if (wasmInitPromise !== null) {
    return wasmInitPromise;
  }

  wasmInitPromise = (async () => {
    const isNodeJs =
      typeof process !== "undefined" &&
      process.versions != null &&
      process.versions.node != null;

    if (isNodeJs) {
      // Node.js (vitest) — synchronous load from disk
      // initSync() does not fetch over HTTP; it reads bytes directly.
      // This avoids the async init problem in vitest's synchronous test runner.
      const { readFileSync } = await import("node:fs");
      const { resolve, dirname } = await import("node:path");
      const { fileURLToPath } = await import("node:url");
      const __filename = fileURLToPath(import.meta.url);
      const __dirname = dirname(__filename);
      const wasmPath = resolve(
        __dirname,
        "pkg/taxklaro_engine_bg.wasm"
      );
      const wasmBytes = readFileSync(wasmPath);
      const { initSync } = await import("./pkg/taxklaro_engine.js");
      initSync({ module: wasmBytes });
    } else {
      // Browser — async fetch of .wasm file
      // Vite + vite-plugin-wasm handles the URL resolution automatically.
      // The default import IS the init function.
      const { default: init } = await import("./pkg/taxklaro_engine.js");
      await init();
    }
  })();

  return wasmInitPromise;
}

// ── Error Classes ─────────────────────────────────────────────────────────────

/**
 * Thrown when the engine rejects user-provided input.
 * Has a code (ERR_* or VAL-*), userMessage for display, and optional field name.
 */
export class ValidationError extends Error {
  public readonly code: string;
  public readonly userMessage: string;
  public readonly field: string | null;

  constructor(code: string, userMessage: string, field: string | null) {
    super(userMessage);
    this.name = "ValidationError";
    this.code = code;
    this.userMessage = userMessage;
    this.field = field;
  }
}

/**
 * Thrown when the engine has an internal bug (assertion error) or
 * the WASM module itself fails to parse/serialize.
 * Should be logged to Sentry; show generic error message to user.
 */
export class ComputeError extends Error {
  public readonly type: "INTERNAL" | "PARSE";

  constructor(type: "INTERNAL" | "PARSE", message: string) {
    super(message);
    this.name = "ComputeError";
    this.type = type;
  }
}

// ── WasmResult Type (internal to bridge) ─────────────────────────────────────

// These types mirror the Rust WasmResult enum wire format.
// They are NOT exported — callers receive TaxComputationResult or an error.

interface WasmSuccess {
  status: "ok";
  data: TaxComputationResult;
}

interface WasmErrorItem {
  code: string;
  message: string;
  field: string | null;
  severity: "HARD_ERROR" | "ASSERTION_ERROR";
  userMessage: string;
}

interface WasmErrorResult {
  status: "error";
  errors: WasmErrorItem[];
}

type WasmResult = WasmSuccess | WasmErrorResult;

// ── ValidationOnlyResult wire format (internal) ───────────────────────────────

interface WasmValidationSuccess {
  status: "ok";
  data: ValidationOnlyResult;
}

type WasmValidationResult = WasmValidationSuccess | WasmErrorResult;

// ── Public API ────────────────────────────────────────────────────────────────

/**
 * Compute full tax liability for a given TaxpayerInput.
 *
 * @throws ValidationError — if the engine rejects user input (show to user)
 * @throws ComputeError — if there's an internal engine bug (log to Sentry, show generic message)
 */
export async function computeTax(
  input: TaxpayerInput
): Promise<TaxComputationResult> {
  await ensureWasmInitialized();
  const { compute_json } = await import("./pkg/taxklaro_engine.js");

  let rawJson: string;
  try {
    rawJson = compute_json(JSON.stringify(input));
  } catch (err) {
    // Only thrown on JSON parse failure of input or internal serialize failure
    throw new ComputeError("PARSE", String(err));
  }

  const result: WasmResult = JSON.parse(rawJson);

  if (result.status === "error") {
    // Return first error — the engine short-circuits on first validation failure
    const firstError = result.errors[0];
    if (firstError.severity === "ASSERTION_ERROR") {
      // Engine bug — should never happen in production
      // Log to Sentry in the calling layer (useTaxBridge hook)
      throw new ComputeError("INTERNAL", firstError.message);
    } else {
      throw new ValidationError(
        firstError.code,
        firstError.userMessage,
        firstError.field
      );
    }
  }

  return result.data;
}

/**
 * Validate input only (runs PL-01 and PL-04, no full computation).
 * Used for per-step wizard validation.
 *
 * Returns ValidationOnlyResult on success.
 * Does NOT throw — validation errors are embedded in the returned result.
 */
export async function validateInput(
  input: Partial<TaxpayerInput>
): Promise<ValidationOnlyResult> {
  await ensureWasmInitialized();
  const { validate_json } = await import("./pkg/taxklaro_engine.js");

  let rawJson: string;
  try {
    rawJson = validate_json(JSON.stringify(input));
  } catch (err) {
    // JSON parse failure — treat as all-fields-invalid
    throw new ComputeError("PARSE", String(err));
  }

  const result: WasmValidationResult = JSON.parse(rawJson);

  if (result.status === "error") {
    // This should not happen for validate_json — it always returns ok with embedded errors
    // If it does, it means the engine itself crashed
    const firstError = result.errors[0];
    throw new ComputeError("INTERNAL", firstError.message);
  }

  return result.data;
}
```

## 5. Vitest Setup File

**Location:** `frontend/src/test-setup.ts`

```typescript
/**
 * Vitest global setup file.
 * Referenced in vite.config.ts test.setupFiles.
 *
 * Performs synchronous WASM initialization before any tests run.
 * This avoids the "WASM not initialized" error in tests that call computeTax().
 *
 * Individual test files do NOT need to call any init function.
 * The bridge.ts module's ensureWasmInitialized() detects Node.js and uses initSync().
 */

import { readFileSync } from "node:fs";
import { resolve, dirname } from "node:path";
import { fileURLToPath } from "node:url";

// Pre-initialize WASM synchronously before tests run.
// This mirrors what bridge.ts does on first call, but does it globally
// so there's no async timing issue in test setup.
const __filename = fileURLToPath(import.meta.url);
const __dirname = dirname(__filename);
const wasmPath = resolve(
  __dirname,
  "wasm/pkg/taxklaro_engine_bg.wasm"
);

// initSync is synchronous — no await needed
// This is safe in vitest's setup file context
import { initSync } from "./wasm/pkg/taxklaro_engine.js";
const wasmBytes = readFileSync(wasmPath);
initSync({ module: wasmBytes });
```

**Important note:** The vitest setup file uses a top-level `initSync` call (not inside an async function). This is intentional — setup files run before any tests and can be synchronous. The `readFileSync` call is safe here.

## 6. Race Condition Prevention

The `ensureWasmInitialized()` pattern prevents the following race condition:

```
Thread 1: computeTax() called → wasmInitPromise is null → starts init
Thread 2: computeTax() called simultaneously → wasmInitPromise is null → starts SECOND init
Result: Two concurrent WASM loads → undefined behavior
```

**The fix:** Store the initialization promise before it resolves:

```typescript
// CORRECT — stores the promise immediately, before awaiting
wasmInitPromise = (async () => { ... })();

// WRONG — would allow race condition
// (another caller could read null before this resolves)
wasmInitPromise = await (async () => { ... })();
```

**Secondary initialization in bridge.ts:** The `compute_json` and `validate_json` functions are imported dynamically inside `computeTax()` and `validateInput()` using `await import(...)`. This is done AFTER `ensureWasmInitialized()` resolves. The dynamic import is cached by the JavaScript module system — it only fetches/evaluates once regardless of how many times it's called.

**Alternative pattern (simpler for single-threaded browser):** Import the WASM module statically at the top of `bridge.ts`. This works in the browser but breaks in Node.js (vitest) because the static import triggers WASM initialization before the test setup file has run. The dynamic import pattern is safer.

```toml
[package]

edition = "2021"

[lib]
crate-type = ["cdylib", "rlib"]
# cdylib: required for WASM output (wasm-pack)
# rlib: required for Rust unit tests (cargo test)

[dependencies]
serde = { version = "1", features = ["derive"] }
serde_json = "1"
serde_with = { version = "3", features = ["macros"] }
rust_decimal = { version = "1", features = ["serde-with-str"] }
chrono = { version = "0.4", features = ["serde"] }
wasm-bindgen = "0.2"

[dev-dependencies]
wasm-bindgen-test = "0.3"

[profile.release]
opt-level = "z"    # Minimize WASM binary size
lto = true
codegen-units = 1
panic = "abort"    # Required for WASM — unwind not supported
```

**Critical:** `panic = "abort"` in `[profile.release]` is required for WASM compilation. Rust's default panic strategy (`unwind`) is not supported by WASM and will cause a linker error.

## 12. `useTaxBridge` Hook (React)

**Location:** `frontend/src/hooks/useTaxBridge.ts`

```typescript
import { useState, useCallback } from "react";
import {
  computeTax,
  validateInput,
  ValidationError,
  ComputeError,
} from "@/wasm/bridge";
import type { TaxpayerInput } from "@/types/engine-input";
import type {
  TaxComputationResult,
  ValidationOnlyResult,
} from "@/types/engine-output";
import * as Sentry from "@sentry/react";

interface UseTaxBridgeReturn {
  compute: (input: TaxpayerInput) => Promise<void>;
  validate: (input: Partial<TaxpayerInput>) => Promise<ValidationOnlyResult | null>;
  result: TaxComputationResult | null;
  error: ValidationError | ComputeError | null;
  isComputing: boolean;
  clear: () => void;
}

export function useTaxBridge(): UseTaxBridgeReturn {
  const [result, setResult] = useState<TaxComputationResult | null>(null);
  const [error, setError] = useState<ValidationError | ComputeError | null>(null);
  const [isComputing, setIsComputing] = useState(false);

  const compute = useCallback(async (input: TaxpayerInput) => {
    setIsComputing(true);
    setError(null);
    try {
      const res = await computeTax(input);
      setResult(res);
    } catch (err) {
      if (err instanceof ComputeError && err.type === "INTERNAL") {
        // Log engine bugs to Sentry — these should never happen in production
        Sentry.captureException(err, {
          extra: { inputSummary: { taxYear: input.taxYear, taxpayerType: input.taxpayerType } },
        });
      }
      setError(err instanceof Error ? (err as ValidationError | ComputeError) : new ComputeError("INTERNAL", String(err)));
    } finally {
      setIsComputing(false);
    }
  }, []);

  const validate = useCallback(
    async (input: Partial<TaxpayerInput>): Promise<ValidationOnlyResult | null> => {
      try {
        return await validateInput(input);
      } catch (err) {
        // Validation errors are returned in the result, not thrown
        // A throw here means engine bug — log and return null
        if (err instanceof ComputeError) {
          Sentry.captureException(err);
        }
        return null;
      }
    },
    []
  );

  const clear = useCallback(() => {
    setResult(null);
    setError(null);
  }, []);

  return { compute, validate, result, error, isComputing, clear };
}
```

## 14. Potential Failure Modes and Mitigations

| Failure Mode | Cause | Mitigation |
|---|---|---|
| White page on prod build | `vite-plugin-top-level-await` breaks prototype chain | Phase 7 smoke test: `npm run build && npx serve dist`, then Playwright verify computation works |
| "WASM not initialized" in vitest | Test file calls `computeTax()` before `test-setup.ts` runs | Ensure `setupFiles: ["./src/test-setup.ts"]` is in vitest config and test-setup.ts calls initSync |
| Race condition: double init | Two concurrent `computeTax()` calls before init | Promise-caching in `ensureWasmInitialized()` — promise stored before await |
| WASM binary not found in test | `resolve(__dirname, "pkg/taxklaro_engine_bg.wasm")` path wrong | Use `import.meta.url` + `fileURLToPath` to compute absolute path from the file's location |
| TypeScript can't resolve pkg/ | Missing or wrong `.d.ts` in pkg/ | Run `wasm-pack build` to regenerate. The generated `.d.ts` must match the `#[wasm_bindgen]` exports |
| `panic = "abort"` not set | WASM link error: "unwind not supported" | Set in `[profile.release]` in `Cargo.toml`. Not required for debug builds (cargo test) |
| `--target bundler` instead of `web` | Vite can't resolve WASM import | Always use `--target web` with `vite-plugin-wasm` |
| `serde-float` feature on rust_decimal | Decimal serializes as JSON float, losing precision | Use `serde-with-str` feature (Cargo.toml dependency is explicit about this) |
