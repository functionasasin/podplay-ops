## POST-MORTEM: Premature convergence (caught manually 2026-03-06)

### What was missed
All route page files in `src/routes/` were still one-liner stubs returning placeholder
text like `<div>Dashboard</div>` or `<div>New Computation</div>`. The forward loop
built all 29 wizard/results components but never wired them into the route pages that
actually render on screen. The app loaded but showed only placeholder text everywhere.

### Root cause: Narrow convergence criteria
The stub detection command only caught components that:
1. Were < 25 lines
2. Contained `return null`

Route page stubs did NOT match because they returned `<div>text</div>` instead of
`null`. A `<div>` with hardcoded placeholder text is functionally a stub, but the
detection regex didn't recognize it as one.

### What should have been checked
The convergence criteria should have included:
1. **Route-level integration check**: Do route pages import and render the components
   that were built? A `grep -L "import.*components/"` on route files would have caught
   this instantly.
2. **Functional smoke test**: Does navigating to `/computations/new` actually show a
   wizard? The loop never tested the app from a user's perspective.
3. **Broader stub definition**: Any page component whose entire render output is a
   single `<div>` with only text content is a stub, regardless of whether it returns
   `null`.

### Fix applied
Manually wired up:
- `/` (index) — Dashboard with recent computations, auth landing
- `/computations/new` — Full wizard with all 18 steps + progress bar + nav controls
- `/computations/$compId` — Computation detail with ResultsView

### Lesson for future loops
**Components are not features.** Building components in isolation without verifying
they are reachable from the router means the app is still a shell. Convergence checks
must verify end-to-end integration, not just component-level completeness.
