**Aspect:** spec-shareable-links
**Wave:** 2 — Per-Feature Specification
**Date:** 2026-03-01
**Reads:** spec-auth-persistence, codebase-audit
**Depends on:** spec-auth-persistence (complete)

## 1. Overview

Philippine estate lawyers frequently need to share preliminary inheritance computations with clients, co-counsel, or heirs without giving them full account access. Today the only path is manual: export JSON, copy-paste narratives, or screenshot the screen. This feature generates a permanent, read-only URL for any saved case that any recipient can open in a browser—no account required.

**What lawyers get:**
- One-click URL generation per case (UUID-based token, not sequential IDs)
- QR code generated client-side for printing on physical handout packets
- Read-only view that mirrors the full ResultsView but strips all editing controls
- Firm branding (name, counsel name, address) included in the shared view
- Instant revocation — once revoked, the URL permanently stops working
- Privacy warning surfaced prominently before sharing so lawyers can make an informed decision

**Business model note:** Shareable links require the lawyer to be authenticated (to have a saved case). This is a premium-tier feature that works only once `spec-auth-persistence` is active. Anonymous (ephemeral) computations cannot be shared via link — the lawyer must save the case first, gating this feature behind registration.

## 2. Data Model

### 2.1 Existing Columns (already defined in spec-auth-persistence)

The `cases` table already carries the two columns needed for sharing. No schema migration is required beyond what `spec-auth-persistence` ships:

```sql
-- Already in cases table (spec-auth-persistence §2.1):
share_token   TEXT UNIQUE,           -- NULL = never shared or revoked
share_enabled BOOLEAN NOT NULL DEFAULT FALSE,

-- Already defined index (spec-auth-persistence §2.1):
CREATE INDEX idx_cases_share_token ON cases(share_token)
  WHERE share_token IS NOT NULL;

-- Already defined RLS policy (spec-auth-persistence §2.1):
CREATE POLICY "cases_select_shared" ON cases
  FOR SELECT USING (share_enabled = TRUE AND share_token IS NOT NULL);
```

**Token format:** UUID v4 generated by Postgres `gen_random_uuid()` — 36 characters, 122 bits of entropy. Guessing a valid token requires ~2^122 attempts. This is sufficient for non-adversarial sharing (disgruntled ex-colleague brute-forcing client data is out of scope; bearer-token URL access is the accepted threat model for shared documents).

### 2.2 Supabase RPC — get_shared_case

The shared view route must load both the case AND the owner's firm profile in a single authenticated-server call (because the RLS on `user_profiles` blocks anonymous reads). A SECURITY DEFINER function bypasses that restriction while still enforcing the share_enabled check:

```sql
-- supabase/migrations/004_shared_case_rpc.sql

CREATE OR REPLACE FUNCTION get_shared_case(p_token TEXT)
RETURNS JSON
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_case        cases%ROWTYPE;
  v_profile     user_profiles%ROWTYPE;
BEGIN
  -- Enforce: only active shared cases
  SELECT * INTO v_case
  FROM cases
  WHERE share_token = p_token
    AND share_enabled = TRUE;

  IF NOT FOUND THEN
    RETURN NULL;   -- caller shows 404
  END IF;

  -- Fetch firm profile of the case owner (bypasses user_profiles RLS)
  SELECT * INTO v_profile
  FROM user_profiles
  WHERE id = v_case.user_id;

  RETURN json_build_object(
    'case',         row_to_json(v_case),
    'firm_profile', CASE WHEN v_profile.id IS NOT NULL
                         THEN row_to_json(v_profile)
                         ELSE NULL
                    END
  );
END;
$$;

-- Grant execute to the anon role (unauthenticated browser requests use this role)
GRANT EXECUTE ON FUNCTION get_shared_case(TEXT) TO anon;
```

**Why SECURITY DEFINER is safe here:**
- The function itself enforces `share_enabled = TRUE` before returning any data.
- The anon role can only call this function — it cannot directly SELECT from `cases` or `user_profiles` for rows that don't pass RLS.
- The function returns a single JSON blob; it does not expose a general query interface.

## 4. API / Data Layer

### 4.1 Generate Share Link

```typescript
// lib/sharing.ts
import { supabase } from './supabase'

export interface GenerateLinkResult {
  token: string
  url: string
  error: Error | null
}

export async function generateShareLink(
  caseId: string
): Promise<GenerateLinkResult> {
  // Generate a UUID v4 token via Postgres (avoids any client-side UUID collisions)
  const { data, error } = await supabase
    .from('cases')
    .update({
      share_token: crypto.randomUUID(),  // browser native; no library needed
      share_enabled: true
    })
    .eq('id', caseId)
    .select('share_token')
    .single()

  if (error || !data?.share_token) {
    return { token: '', url: '', error: error as Error ?? new Error('Token generation failed') }
  }

  const token = data.share_token
  const url = `${window.location.origin}/share/${token}`
  return { token, url, error: null }
}
```

**Why `crypto.randomUUID()` client-side:**
`crypto.randomUUID()` is available in all modern browsers (Chrome 92+, Firefox 95+, Safari 15.4+) and produces a cryptographically secure v4 UUID. The `UNIQUE` constraint on `share_token` will reject the astronomically unlikely collision. The alternative is a Postgres `gen_random_uuid()` via RPC, but the client-side call saves a round-trip. Both produce identical entropy.

### 4.2 Revoke Share Link

```typescript
// lib/sharing.ts (continued)

export async function revokeShareLink(caseId: string): Promise<Error | null> {
  const { error } = await supabase
    .from('cases')
    .update({
      share_token: null,
      share_enabled: false
    })
    .eq('id', caseId)

  return error as Error | null
}
```

Revoking sets both `share_token = NULL` and `share_enabled = FALSE`. Nulling the token means the old URL can never be re-enabled — a new `crypto.randomUUID()` is generated on the next `generateShareLink()` call. This is intentional: revocation must be definitive and irreversible from the old URL.

### 4.3 Load Shared Case (Public/Anon)

```typescript
// lib/sharing.ts (continued)

import type { SharedCasePayload } from '@/types/sharing'

export async function loadSharedCase(
  token: string
): Promise<SharedCasePayload | null> {
  const { data, error } = await supabase.rpc('get_shared_case', {
    p_token: token
  })

  if (error || !data) return null

  // The RPC returns a JSON object; Supabase returns it as a parsed JS object
  return data as SharedCasePayload
}
```

**RLS note:** `loadSharedCase` uses the anon key (no user session). The SECURITY DEFINER function on Postgres handles the cross-table read (cases + user_profiles). No client-side auth is needed.

### 4.4 React Hook — useSharedCase

```typescript
// hooks/useSharedCase.ts
import { useEffect, useState } from 'react'
import { loadSharedCase } from '@/lib/sharing'
import type { SharedCasePayload } from '@/types/sharing'

export type SharedCaseState =
  | { status: 'loading' }
  | { status: 'loaded'; payload: SharedCasePayload }
  | { status: 'not-found' }
  | { status: 'no-output' }
  | { status: 'error'; message: string }

export function useSharedCase(token: string): SharedCaseState {
  const [state, setState] = useState<SharedCaseState>({ status: 'loading' })

  useEffect(() => {
    if (!token) {
      setState({ status: 'not-found' })
      return
    }

    loadSharedCase(token)
      .then(payload => {
        if (!payload) {
          setState({ status: 'not-found' })
        } else if (!payload.case.output_json) {
          setState({ status: 'no-output' })
        } else {
          setState({ status: 'loaded', payload })
        }
      })
      .catch(err => setState({ status: 'error', message: String(err) }))
  }, [token])

  return state
}
```

## 6. Edge Cases

### 6.1 Share Token Collision

UUID v4 has 2^122 ≈ 5.3 × 10^36 possible values. At 1 million cases, the probability of any collision is ~1.9 × 10^{-28}. The `UNIQUE` constraint on `share_token` catches this and returns a Postgres unique-violation error (`23505`). The client should retry with a new `crypto.randomUUID()`:

```typescript
export async function generateShareLink(caseId: string): Promise<GenerateLinkResult> {
  for (let attempt = 0; attempt < 3; attempt++) {
    const token = crypto.randomUUID()
    const { data, error } = await supabase
      .from('cases')
      .update({ share_token: token, share_enabled: true })
      .eq('id', caseId)
      .select('share_token')
      .single()

    if (!error && data?.share_token) {
      return { token: data.share_token, url: `${window.location.origin}/share/${data.share_token}`, error: null }
    }

    // Only retry on unique constraint violation
    const pgError = error as { code?: string } | null
    if (pgError?.code !== '23505') {
      return { token: '', url: '', error: error as Error }
    }
    // else: loop and try again
  }

  return { token: '', url: '', error: new Error('Could not generate a unique share token after 3 attempts') }
}
```

In practice this retry logic will never execute — it exists purely as a safety net.

### 6.2 Edge Cases Table

| Scenario | Handling |
|----------|----------|
| Token in URL doesn't exist | `get_shared_case` returns NULL → `useSharedCase` → `status: 'not-found'` → "Link not available" page |
| Token exists but `share_enabled = FALSE` (revoked) | Same: function returns NULL (condition `share_enabled = TRUE` fails) → "Link not available" page |
| `output_json` is null (shared draft) | `useSharedCase` → `status: 'no-output'` → "Case not computed yet" message |
| `input_json` is null (corrupt record) | `output_json` also null in this case → same as above |
| Network error loading shared case | `useSharedCase` → `status: 'error'` → "Unable to load case. Check your connection." |
| Lawyer tries to share unsaved case | Share button is hidden (no `caseId`) — not reachable |
| Lawyer shares finalized case | Fully supported — shared view shows "Finalized" badge from `cases.status` |
| Lawyer shares then edits case after sharing | New computation updates `output_json` in DB; shared URL now shows updated output. No notification to viewers (by design — no view tracking) |
| Viewer opens shared link after owner deletes case | `ON DELETE CASCADE` removes the cases row → `get_shared_case` returns NULL → "Link not available" page |
| Two viewers open shared link simultaneously | Stateless read — no conflict |
| Viewer tries to modify URL (path traversal) | The `/share/:token` route only passes the token to `get_shared_case` — no direct table access, no SQL injection risk (parameterized query) |
| Very long case title in QR PNG filename | Filename is truncated to 40 characters: `.slice(0, 40)` before sanitization |
| `crypto.randomUUID()` not available (old browser) | Chrome 92+, Firefox 95+, Safari 15.4+ all support it. If not, fallback: `Math.random()` based UUID is NOT acceptable. Display: "Your browser does not support secure link generation. Please update your browser." |

### 6.3 Privacy Considerations

The Share dialog surfaces a prominent warning before the first link generation (§3.2). This is intentional — the lawyer must acknowledge the data exposure before the link is created. The warning is shown every time the dialog opens (not dismissed via "don't show again"), because:

1. Philippine data privacy law (Republic Act No. 10173, Data Privacy Act of 2012) requires informing data subjects about processing.
2. Bar ethics rules on client confidentiality require lawyers to assess risk before transmitting confidential information.
3. A dismissed warning could create liability if the lawyer claims they didn't know.

The warning is non-blocking (no checkbox required) but is always displayed.

## 8. Acceptance Criteria

### Share Link Generation

- [ ] "Share" button is visible in ActionsBar only when user is authenticated AND a `caseId` exists
- [ ] Clicking "Share" opens the ShareDialog
- [ ] ShareDialog shows the privacy warning every time it opens
- [ ] Clicking "Generate Share Link" calls `generateShareLink(caseId)` and shows the URL within 1 second
- [ ] Generated URL format is `{origin}/share/{uuid-v4}` (36-character UUID)
- [ ] URL is displayed in a read-only text input
- [ ] "Copy" button copies URL to clipboard; button label changes to "Copied!" for 2 seconds
- [ ] QR code renders for the generated URL
- [ ] "Download QR as PNG" downloads a PNG file named `share-qr-{case-title-slug}.png`
- [ ] QR PNG is 160×160px and correctly encodes the URL (verify with a QR scanner)
- [ ] Link status shows "● Active" when `share_enabled = TRUE`
- [ ] Closing and reopening the ShareDialog still shows the active link (token persisted in DB)

### Share Link Revocation

- [ ] "Revoke Link" button opens confirmation dialog
- [ ] Clicking "Revoke Link" in confirmation calls `revokeShareLink(caseId)`
- [ ] After revoke: dialog returns to initial "not-shared" state with "Link revoked" confirmation message
- [ ] Revoked URL (`/share/{old-token}`) shows "Link not available" page (verified by navigating to old URL)
- [ ] New "Generate Share Link" call creates a different UUID (not the old one)
- [ ] Canceling the revoke confirmation leaves the link active

### Read-Only Shared View

- [ ] Navigating to `/share/{valid-token}` with no auth session shows the case results
- [ ] Firm name, counsel name, and address from `user_profiles` appear in the page header
- [ ] If firm profile is null, header shows "Philippine Inheritance Calculator" (no crash)
- [ ] Page shows: scenario badge, succession type, estate value, distribution table, narratives, warnings, computation log
- [ ] All 7 distribution layout variants render correctly in shared view (same as case owner view)
- [ ] "Edit Input", "Save", "Share", "Finalize" buttons are absent from ActionsBar
- [ ] "Export JSON", "Export PDF", "Print" buttons are present
- [ ] Footer shows disclaimer text including "read-only shared view" and generation date
- [ ] Navigating to `/share/{invalid-or-revoked-token}` shows "Link not available" page
- [ ] "Link not available" page links back to `/` (calculator home)
- [ ] Case with `output_json = null` shows "Case not computed yet" message (not a crash)

### Security

- [ ] Direct SELECT on `cases` with `share_enabled = FALSE` and a token returns nothing for anon role (verify via Supabase REST with anon key)
- [ ] `get_shared_case` with a non-existent token returns null (not an error with a real case payload)
- [ ] A user cannot view another user's case via the share endpoint if that case has `share_enabled = FALSE`
- [ ] The `get_shared_case` function is only `GRANT EXECUTE` to `anon` — not to `authenticated` (authenticated users use normal case read via RLS)
- [ ] No `service_role` key exposed in client bundle (verify with `grep -r "service_role" src/`)

### QR Code

- [ ] QR code is scannable on a physical printout at ≥2cm width (test at 160px ÷ 96dpi ≈ 4.2cm — should pass)
- [ ] QR code level is "M" (medium error correction — tolerant of minor print degradation)
- [ ] QR code updates if the dialog is closed, link revoked, and a new link generated (new token = new QR)
