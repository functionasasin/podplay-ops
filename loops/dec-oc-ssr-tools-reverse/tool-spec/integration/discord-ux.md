## Overview

This file specifies how SSR consumer panel results render in Discord. Discord has a hard 2000-character message limit per message. The bot uses threads for long-form output. The w2-tool-panel-results analysis established the output format sizing and rendering constraints documented here. The w3-discord-ux aspect will fill in the full interactive UX flow (progress indicators, panel creation acknowledgment, run progress, etc.).

## 2. Panel Results Rendering — Single Run

The bot posts the XML tool output and Claude synthesizes it into a readable Discord message. The Claude agent converts the `<panel-results>` XML into natural-language Discord markdown. The following shows the natural-language equivalent the agent should produce.

### 2.1 Summary Format (3 dimensions) — Example Discord Message

```
**Panel Results: "Version A — emotional approach"** · Filipino Moms 30-45

**Stimulus** (social caption): *"Ang sarap, tipid pa! Lucky Me na naman kasama ko sa hapag-kainan."*

| Dimension | Mean | Mode | Distribution |
|-----------|------|------|-------------|
| Purchase Intent | 3.85/5 | 4 | ▁▂▅████▄ (1:1 2:2 3:4 4:9 5:4) |
| Message Clarity | 4.40/5 | 5 | ▁▁▂████▉▉ (1:0 2:1 3:2 4:8 5:9) |
| Personal Relevance | 4.10/5 | 4 | ▁▂▅████▅▅ (1:0 2:2 3:4 4:9 5:5) |

**Highlights**
> *"I'd definitely grab another pack — we always have Lucky Me at home."* — Maria Santos ✅
> *"I don't buy Lucky Me anymore. I switched to a cheaper brand."* — Celine Vera ❌

20 personas scored · Run ID: `f9e1c2d3`
```

**Notes on the above format**:
- `▁▂▅████▄` distribution bars are Unicode block characters — Claude generates these from the distribution data to give a visual sense of skew at a glance.
- The actual bot output will be generated by Claude interpreting the XML — the exact markdown format above is a target, not enforced output.
- Run ID is shown truncated to 8 chars for readability; the full ID is in the hint for copy-paste.
- Highlights show ✅ for positive valence, ❌ for negative.

### 2.2 Detailed Format — Thread Delivery

For `response_format="detailed"` with 3+ dimensions, the bot should:
1. Post a short acknowledgment message in the channel: *"Results are detailed — posting in a thread."*
2. Create or use an existing thread for this panel
3. Post full results in the thread (multiple messages if needed)

Each dimension's detailed output in a thread:

```
**Purchase Intent** · Mean: 3.85/5 · Std: 0.91 · Mode: 4 · 95% CI: [3.43–4.27]
Distribution: 1▌1 2▌▌2 3▌▌▌▌4 4▌▌▌▌▌▌▌▌▌9 5▌▌▌▌4

Positives:
> *"I'd definitely grab another pack — we always have Lucky Me at home."* — Maria Santos
> *"I'd pick this up on my next grocery run."* — Tess Aguilar

Negatives:
> *"I don't buy Lucky Me anymore. I switched to a cheaper brand."* — Celine Vera
> *"My doctor said to cut down on instant noodles."* — Rosa Dela Paz

Neutral:
> *"I buy it sometimes when I'm in a hurry, not really something I choose over fresh food."* — Ana Bautista
```

### 2.3 Raw Format — Thread Delivery, Multiple Messages

Raw format with 20+ personas will exceed 2000 chars per message. The bot should:
1. Post aggregated summary first (always fits in ~1 message)
2. Then post persona-by-persona breakdown across multiple messages, batched by ~1800 chars
3. Begin each batch with a continuation header: *"[Persona responses 1–7 of 20]"*, *"[8–14 of 20]"*, etc.

## 3. Comparison Results Rendering

### 3.1 Summary Comparison — Example Discord Message

```
**Panel Comparison: A vs B**

**A:** "Version A — emotional approach" · social caption
**B:** "Version B — practical approach" · social caption

| Dimension | A Mean | B Mean | Delta |
|-----------|--------|--------|-------|
| Purchase Intent | 3.85 | 4.12 | +0.27 ↑ ✨ |
| Message Clarity | 4.40 | 4.55 | +0.15 ↑ |
| Personal Relevance | 4.10 | 3.88 | -0.22 ↓ |

✨ = statistically notable (non-overlapping 95% confidence intervals)

**Bottom line:** Version B outperforms on purchase intent and clarity, but Version A resonates more personally with this panel. For driving purchase, Version B wins.

20 personas each · Run A: `f9e1c2d3` · Run B: `a4b5c6d7`
```

**Notes**:
- `✨` marks significant differences (non-overlapping CIs) — this is the Discord-rendered version of the `*` marker in the XML.
- The "Bottom line" synthesis is generated by Claude from the XML comparison data — the agent should summarize the comparison in 1–2 sentences.
- Both run IDs are shown (truncated) for reference.

Detailed comparison per dimension in a thread:

## 4. Character Budget for Summary Format

To ensure `summary` format fits in a single Discord message (2000 char limit), the following character budget applies:

| Component | Max Characters |
|-----------|---------------|
| Header (panel name, run label, stimulus preview) | ~200 chars |
| Table row per dimension (name, mean, mode, dist) | ~120 chars each |
| 2 highlight quotes × 200 chars each per dimension | ~400 chars per dimension |
| Footer (personas scored, run ID) | ~60 chars |
| **Budget for 3 dimensions** | ~200 + 3×120 + 3×400 + 60 = **1,820 chars** |
| **Budget for 5 dimensions** | ~200 + 5×120 + 5×400 + 60 = **2,860 chars** (thread needed) |

**Conclusion**: Summary format is safe for up to 4 dimensions in a single message. For 5+ dimensions, the bot should post in a thread. This threshold applies regardless of panel size (quotes are truncated to 200 chars).

## 5. Progress Indicators — Full Specification

Daimon uses the Claude Agent SDK in streaming mode. **Text Claude generates before calling a tool is streamed to Discord in real time**, before the tool call begins executing. This is the primary mechanism for progress indication — Claude narrates what it is about to do, the text appears in Discord immediately, and then the tool runs (silently, from the user's perspective) for several seconds.

There is **no message-editing** or "typing..." indicator in the current Daimon architecture. Progress is communicated entirely through Claude's streamed text before each tool call and after each tool completes.

### 5.1 Standard Single-Turn Flow: Create + Run

**User message:**
```
@Daimon test this tagline with Filipino moms 30-45: "Ang sarap ng tipid!"
```

**Bot streaming output (before first tool call):**
```
Sure! Let me set up a consumer panel for Filipino moms ages 30–45 and run your tagline through it. Building the panel first — I'll have results in about 20–30 seconds.
```
*(This text appears immediately in Discord, before any tool calls begin.)*

**Then Claude calls (silently):**
1. `ssr_panel_list` (~0.5s, DB read — checks for existing matching panel)
2. `ssr_panel_create` (~10–20s, 20 concurrent Haiku calls)

**Bot streaming output (between panel create and panel run):**
```
Panel ready — 20 personas across Quezon City, Davao, Cebu, and Metro Manila. Running the tagline now...
```

**Then Claude calls (silently):**
3. `ssr_panel_run` (~8–15s, 20 concurrent Haiku calls + 1 OpenAI embedding batch)

**Bot final output (after all tools complete):**
*(full results — see §2 above)*

**Total elapsed time visible to user**: The first text appears within 1–2 seconds of the user's message. Results appear 20–40 seconds later. The intermediate "Panel ready... running now..." message appears ~10–20 seconds after the initial message.

### 5.2 Pre-Tool Acknowledgment Text — by Tool

The following table specifies the exact text Claude should emit before each SSR tool call. This text is generated by the Claude agent in response to the `<consumer-panel>` system prompt guidance. The actual phrasing will vary, but should match this template semantically.

#### Before `ssr_panel_list` (first call in a session)

No separate pre-tool text needed for `ssr_panel_list` — it runs as part of the preamble before `ssr_panel_create`. Claude does NOT need to tell the user "I'm checking your existing panels" — this is invisible plumbing.

Exception: If the user explicitly asks "what panels do I have?", Claude should say:
```
Here are your saved consumer panels:
```
*(then call `ssr_panel_list` and post the results in the format specified in §7)*

#### Before `ssr_panel_create` (new panel case)

```
Let me build a panel of {panel_size} {demographic_summary} personas — this takes about {estimated_time}.
```

Where:
- `{panel_size}` = the integer panel size (e.g., `20`)
- `{demographic_summary}` = a natural-language description of the demographic (e.g., `Filipino women ages 30–45`, `Gen Z gamers ages 18-24`, `urban professionals in Manila ages 28–35`)
- `{estimated_time}` = `15–20 seconds` for 20 personas, `25–35 seconds` for 50 personas, `8–12 seconds` for 5 personas

Examples:
```
Let me build a panel of 20 Filipino moms ages 30–45 — this takes about 15–20 seconds.
```
```
Building 50 Gen Z gamers ages 18–24 for you. This'll take about 25–35 seconds — grab a coffee.
```

#### Before `ssr_panel_create` (partial success — if discovered after)

No pre-tool text. The partial outcome is communicated in Claude's response AFTER the tool returns. See §13 for partial panel rendering.

#### Before `ssr_panel_run` (standalone, panel already exists)

```
Running your {stimulus_type_label} through the {panel_name} panel ({panel_size} personas) — about {estimated_time}.
```

Where:
- `{stimulus_type_label}` = human-readable label from the table below
- `{panel_name}` = the panel's name
- `{panel_size}` = number of personas
- `{estimated_time}` = `8–12 seconds` for ≤20 personas, `15–25 seconds` for 21–50 personas

Stimulus type labels:
| `stimulus_type` value | Human-readable label |
|----------------------|---------------------|
| `ad_copy` | ad copy |
| `tagline` | tagline |
| `headline` | headline |
| `social_caption` | social caption |
| `product_concept` | product concept |
| `positioning_statement` | positioning statement |
| `campaign_brief` | campaign brief |
| `influencer_pitch` | influencer pitch |
| `pricing_message` | pricing message |
| `packaging_concept` | packaging concept |

Examples:
```
Running your tagline through the Filipino Moms 30-45 panel (20 personas) — about 8–12 seconds.
```
```
Running your product concept through the Gen Z Filipinas 18-24 panel (25 personas) — about 12–18 seconds.
```

#### Before `ssr_panel_run` (chained immediately after `ssr_panel_create`)

```
Panel ready with {actual_size} personas. Running your {stimulus_type_label} through all of them now...
```

Example:
```
Panel ready with 20 personas. Running your tagline through all of them now...
```

#### Before `ssr_panel_results` (standalone retrieval)

No pre-tool text needed — results appear in <1 second (DB read). Claude should just post the results directly.

Exception: if the user asked for `raw` format with a large panel (>20 personas), Claude should say before the call:
```
Fetching raw persona-by-persona results — posting in a thread.
```

#### Before `ssr_panel_results` (comparison mode)

No pre-tool text needed — the comparison is computed in-memory from DB records in <1 second. Claude should post results directly.

#### Before comparison flow (two sequential `ssr_panel_run` calls)

When running two assets for comparison:
```
Running both versions through the panel — this will take about {2x_estimated_time} total.
```

Example:
```
Running both taglines through the Filipino Moms 30-45 panel — this will take about 20–30 seconds total.
```

The user should not see "Running Version A... now running Version B..." as separate messages — Claude should acknowledge the full comparison upfront, then run both tools, then post comparison results.

#### Before `ssr_panel_delete`

```
Deleting panel "{panel_name}" and all its personas and run history.
```

Claude must always narrate panel deletion before executing it, so the user has a chance to read what's about to be deleted (the message is posted instantaneously before the tool call).

### 5.3 Post-Tool Final Text — What Claude Emits After Each Operation

After every SSR operation, Claude synthesizes the tool's XML output into a Discord message. This is the final output the user reads. Format guidelines by operation:

#### After `ssr_panel_create` (no subsequent run)

Claude should post a summary of the created panel in this format:
```
**Panel created: {panel_name}** ({panel_size} personas)

{persona_overview_paragraph}

Panel ID: `{panel_id_truncated}` · Estimated run cost: ~${cost}/run
```

Where `{persona_overview_paragraph}` is a 2-3 sentence Claude-synthesized description of the panel demographic mix (not a list of all 20 personas). Examples:
```
The panel covers a range of Filipino women ages 30–45 from Metro Manila, Cebu, and Davao City. Occupations include teachers, store owners, office workers, and stay-at-home moms. Most are in the middle income bracket with high school or college education.
```

For persona listing (when user explicitly requests "show me the personas"):
*(see §7.3 — Panel Persona Listing)*

#### After `ssr_panel_run` (inline results)

See §2.1 (summary format) and §2.2 (detailed format). Claude generates a markdown table with distributions, highlights, and a 1–2 sentence bottom line.

#### After `ssr_panel_results` (retrieval)

Same format as after `ssr_panel_run` — the user should not perceive a difference between inline results and retrieved results.

#### After `ssr_panel_list`

See §7 — Panel List Rendering.

#### After `ssr_panel_delete`

See §8 — Panel Delete Rendering.

### 5.4 Timing Expectations (Visible to Users)

| Operation | Expected Wall Time | What User Sees |
|-----------|------------------|----------------|
| `ssr_panel_list` | < 0.5s | Invisible (always chained before create/run) |
| `ssr_panel_create` (5 personas) | 3–7s | Pre-tool text → results |
| `ssr_panel_create` (20 personas) | 10–20s | Pre-tool text → [wait] → results |
| `ssr_panel_create` (50 personas) | 25–50s | Pre-tool text → [wait] → results |
| `ssr_panel_run` (5 personas, 1 dim) | 3–5s | Pre-tool text → results |
| `ssr_panel_run` (20 personas, 3 dims) | 8–15s | Pre-tool text → [wait] → results |
| `ssr_panel_run` (50 personas, 10 dims) | 20–40s | Pre-tool text → [wait] → results |
| `ssr_panel_results` (any) | < 1s | No pre-tool text → results |
| `ssr_panel_delete` | < 0.5s | Pre-tool text → confirmation |
| Full create+run flow (20 personas) | 20–35s | Pre-create text → [wait] → inter-tool text → [wait] → results |

**User expectation management**: The pre-tool text's timing estimate must be honest. Use "about 15–20 seconds" not "this will only take a moment" for runs that take >10 seconds.

## 6. Error Message Rendering — All Five Tools

Tool errors from all 5 SSR tools appear in Discord as Claude's natural-language interpretation of the `ToolError` message. Claude should always:
1. Acknowledge what the user was trying to do
2. Explain what went wrong in plain language
3. Offer a concrete next step

### 6.1 `ssr_panel_create` Errors

| ToolError text | Expected Discord rendering |
|---------------|---------------------------|
| `"Database context required for SSR panel tools."` | *"I hit a database issue setting up the panel — this is a system problem. Try again in a moment, or let the bot admin know."* |
| `"Panel generation failed: only N of M personas were generated before an error occurred. Please try again. If the problem persists, reduce panel_size or simplify the demographic constraints."` | *"I hit an error generating the personas — only N out of M were created, which isn't enough to run reliably. Let me try again, or I can try with a smaller panel (maybe {M//2} personas instead of M)?"* |
| `"age_max (X) must be >= age_min (Y)"` | *"Those age bounds don't work — age_max ({X}) needs to be at least age_min ({Y}). Did you mean the range the other way around?"* |
| `"genders must not be an empty list — pass null for any gender"` | *"I need at least one gender in the filter, or leave it open for any gender. Which would you prefer?"* |
| `"income_brackets must not be an empty list — pass null for any income level"` | *"Same issue — if you want to filter by income, give me at least one bracket (low, middle, high, etc.), or leave it open for all income levels."* |
| `"panel_size must be between 5 and 50"` | *"Panel size needs to be between 5 and 50 personas. You asked for {requested} — want me to use {clamped} instead?"* |
| Pydantic validation error on `product_category` | *"I need to know what product category this panel is for — something like 'skincare', 'instant noodles', or 'personal finance apps'. What should I use?"* |

### 6.2 `ssr_panel_run` Errors

| ToolError text | Expected Discord rendering |
|---------------|---------------------------|
| `"Database context required for SSR panel tools."` | *"Database issue — try again in a moment."* |
| `"Panel '{panel_id}' not found. Use ssr_panel_list to see your available panels."` | *"I couldn't find that panel. Let me list your existing panels:"* *(then calls `ssr_panel_list`)* |
| `"Panel '{panel_id}' has no personas. The panel may have failed to generate. Use ssr_panel_create to create a new panel."` | *"That panel exists but has no personas — it may have failed during creation. Want me to create a new one with the same demographics?"* |
| `"stimulus_text is required when stimulus_type is not 'campaign_brief'."` | *"I need the actual marketing asset text to run the panel. What's the ad copy / tagline / concept you want to test?"* |
| `"stimulus_text exceeds maximum length of 2000 characters ({len} chars)."` | *"That's a bit long for a panel stimulus — the max is 2000 characters and yours is {len}. Can you trim it down, or should I test just the key section?"* |
| `"evaluation_dimensions must not be empty."` | *"I need at least one evaluation dimension to measure. The defaults are purchase intent, message clarity, and overall appeal — want me to use those?"* |
| `"Unknown evaluation dimension: '{dim}'. Valid dimensions: {valid_list}."` | *"'{dim}' isn't a recognized dimension. Valid options are: purchase intent, brand favorability, message clarity, emotional response, personal relevance, uniqueness, trust & credibility, value perception, share-worthiness, and overall appeal."* |
| General exception during pipeline | *"Something went wrong during the panel run — {N} of {M} personas were scored before the error. The partial results are below, but I'd recommend re-running for a complete picture."* |

## 7. Panel List Rendering (`ssr_panel_list`)

### 7.1 Standard List Format

When the user asks "what panels do I have?" or "list my panels", Claude calls `ssr_panel_list` and renders the results as follows:

**Zero panels:**
```
You don't have any saved panels yet. To create one, tell me what audience you want to test with — for example: "test this ad copy with Filipino moms ages 30–45".
```

**One panel:**
```
You have 1 saved panel:

**Filipino Moms 30-45** · 20 personas · instant noodles
Last run: 2026-03-01 · 3 runs total · Panel ID: `a1b2c3d4`
```

**Multiple panels (up to 10):**
```
You have {N} saved panels:

1. **Filipino Moms 30-45** · 20 personas · instant noodles
   Last run: 2026-03-01 · 3 runs · `a1b2c3d4`

2. **Gen Z Filipinas 18-24** · 25 personas · skincare
   Last run: 2026-02-28 · 1 run · `b5c6d7e8`

3. **Urban Professionals Manila 28-35** · 15 personas · fintech
   No runs yet · `c9d0e1f2`
```

**Notes on the list format:**
- Truncate panel ID to 8 hex chars for readability
- Show "No runs yet" if `run_count = 0`
- Show "1 run" (singular) or "N runs" (plural)
- Show last run date in `YYYY-MM-DD` format
- List is ordered by most recently created or most recently run (whichever is most recent)
- Maximum 10 panels shown; if more exist: "*(and {N-10} more — ask me to show older panels)*"

### 7.2 Panel List with Run Details

When user asks "show me my panels and their runs" or "what tests have I done?", Claude renders each panel with its run history:

```
**Filipino Moms 30-45** (20 personas · instant noodles · `a1b2c3d4`)

Runs:
• 2026-03-01 "Version B tagline" — purchase_intent 4.12, message_clarity 4.55 · `def45678`
• 2026-02-28 "Version A tagline" — purchase_intent 3.85, message_clarity 4.40 · `abc12345`
• 2026-02-27 "Initial concept" — overall_appeal 3.20 · `xyz98765`
```

This format uses the `include_runs=True` option on `ssr_panel_list`.

### 7.3 Panel Persona Listing

When user explicitly asks "show me the personas in my panel" or "who's in the Filipino moms panel?", Claude calls `ssr_panel_list` with the specific panel ID and renders persona summaries:

```
**Filipino Moms 30-45** — 20 personas:

1. **Maria Santos**, 37 — Quezon City, Teacher. Married, 2 kids. Budget-conscious, trusts church community recommendations.
2. **Liza Reyes**, 42 — Davao City, Store Owner. Widowed, runs sari-sari store. Heavy TikTok user, favors local brands.
3. **Joy Dela Cruz**, 31 — Makati, CSR. Renting with roommate, sends remittances home. Instagram-influenced.
4. **Rosa Bautista**, 44 — Cebu City, Nurse. 3 teenage kids. Health-conscious, skeptical of processed food.
5. **Tess Villanueva**, 38 — Parañaque, Admin Assistant. Husband is OFW. Makes most grocery decisions alone.
```
*(continues through all 20 personas)*

Each persona line follows: `{index}. **{name}**, {age} — {location}, {occupation}. {summary_sentence_1}. {summary_sentence_2}.`

For large panels (>10 personas), Claude should post this in a thread if it would exceed 2000 chars (which it will for 20+ personas).

## 8. Panel Delete Rendering (`ssr_panel_delete`)

### 8.1 Delete Confirmation Message

Before calling `ssr_panel_delete`, Claude narrates what will be deleted (§5.2). After the tool returns:

**Success:**
```
Deleted. Panel "Filipino Moms 30-45" and its 20 personas and 3 run records have been removed.
```

**If the user asks to delete "all my panels"**, Claude should list the panels first and confirm before deleting each:
```
You have 3 panels. I'll delete all of them:
1. Filipino Moms 30-45 (20 personas, 3 runs)
2. Gen Z Filipinas 18-24 (25 personas, 1 run)
3. Urban Professionals Manila 28-35 (15 personas, 0 runs)

Deleting all 3 now...
```
*(Then calls `ssr_panel_delete` three times sequentially)*
```
Done. All 3 panels deleted.
```

### 8.2 Safety Note for Deletion

`ssr_panel_delete` is irreversible — all personas and run results are deleted via `ON DELETE CASCADE`. Claude should **always narrate what will be deleted before calling the tool** (as specified in §5.2), giving the user a chance to read the message (streamed before tool execution) and react.

If the user says "delete everything" in ambiguous context (they might mean "delete the last run" not "delete the panel"), Claude should clarify:
```
Just to confirm — you want me to delete the entire "Filipino Moms 30-45" panel (20 personas, 3 runs), not just the most recent run? This can't be undone.
```

## 9. Thread Creation and Management

### 9.1 When to Use Threads

Discord threads are used when output exceeds the ~2000 character message limit. The decision is made by Claude at the time of posting based on the estimated output length:

| Condition | Delivery Method |
|-----------|----------------|
| Output estimated ≤ 1,800 chars | Single message in channel |
| Output estimated 1,800–2,000 chars | Single message (at risk; Claude may trim if over) |
| Output estimated > 2,000 chars | Thread |
| `response_format="raw"` with any panel size | Always thread |
| `response_format="detailed"` with 3+ dimensions | Always thread |
| Persona listing for panel > 10 personas | Always thread |
| Panel list with run details for 3+ panels | Thread recommended |

### 9.2 Thread Creation

Daimon does not currently have a "create thread and post to it" capability in its tool registry. For the SSR implementation, the bot's response in a channel is a standard Discord message. Discord's native thread creation from a message requires a separate API call.

**Implementation options (in priority order):**

**Option A (recommended): In-channel with message splitting**
The bot posts multiple messages in the channel, each ≤ 2000 chars, prefixed with continuation headers. No thread creation needed.

Format for multi-message output:
```
[Message 1 of 3] **Panel Results: "Version A"** — full results header + first 3 dimensions...
```
```
[Message 2 of 3] ... dimensions 4–6 + highlights...
```
```
[Message 3 of 3] ... dimensions 7–10 + bottom line + run ID footer
```

**Option B (if thread tool added later): Thread-based delivery**
If a `discord_create_thread` tool is added to the bot's tool registry, Claude can create a thread named `Panel: {panel_name}` and post detailed results there. The first channel message becomes:
```
Results are detailed — posting in thread: **Panel: Filipino Moms 30-45**
```

**For the initial implementation, use Option A.** Thread creation is out of scope for the SSR tool set itself.

### 9.3 Multi-Message Splitting Algorithm

When output must be split across messages (Option A):

1. **Determine total sections**: Count the number of dimension blocks to post
2. **Batch into ~1800-char segments**: Group dimension blocks such that each group ≤ 1800 chars
3. **Add continuation headers**: Prefix each message beyond the first with `[continued ↓]` at the top
4. **Preserve section boundaries**: Never split within a dimension block; always break between dimensions
5. **Footer on last message only**: Post run ID, persona count, and bottom line on the final message only

Example split for 5-dimension detailed results:

**Message 1:**
```
**Panel Results: "Lucky Me tagline v1"** · Filipino Moms 30-45 · 20 personas

> "A bit generic — all noodle ads say the same thing." — Tess Aguilar ❌
```

**Message 2:**
```

**Personal Relevance** · Mean: 4.10/5 · Std: 0.80 · Mode: 4 · 95% CI: [3.74–4.46]
...
```

**Message 3 (final):**
```

...
```

## 10. Unicode Distribution Bar Algorithm

The distribution bar format `▁▂▅████▄` displayed in §2 and §3 is constructed from per-rating counts as follows.

### 10.1 Input

The rating distribution is a list of counts per Likert point, e.g., for a 5-point scale:
```python
distribution = {1: 1, 2: 2, 3: 4, 4: 9, 5: 4}  # total: 20
```

### 10.2 Construction Algorithm

```python
# Unicode block characters by height (index 0 = empty, 7 = full block)
BLOCKS = [" ", "▁", "▂", "▃", "▄", "▅", "▆", "▇", "█"]

def build_distribution_bar(distribution: dict[int, int], scale: int = 5) -> str:
    """
    Build a Unicode block-character bar chart from a Likert distribution.

    Args:
        distribution: dict mapping Likert point (1–scale) to count
        scale: number of points in the scale (5 or 7)

    Returns:
        A string of Unicode block characters, one per Likert point.

    Example:
        {1:1, 2:2, 3:4, 4:9, 5:4} → "▁▂▄████▄"
    """
    counts = [distribution.get(i, 0) for i in range(1, scale + 1)]
    max_count = max(counts) if counts else 1
    if max_count == 0:
        return "▁" * scale  # all-zero edge case

    bars = []
    for count in counts:
        # Normalize to 0–8 range (BLOCKS has 9 elements: 0=space, 1=▁, ..., 8=█)
        normalized = round((count / max_count) * 8)
        bars.append(BLOCKS[normalized])

    return "".join(bars)
```

### 10.3 Examples

| Distribution (5-point) | Bar Output | Interpretation |
|------------------------|-----------|----------------|
| `{1:1, 2:2, 3:4, 4:9, 5:4}` | `▁▂▄████▄` | Right-skewed positive (mode 4) |
| `{1:0, 2:1, 3:2, 4:8, 5:9}` | ` ▁▂▇████` | Strongly right-skewed (high clarity) |
| `{1:4, 2:5, 3:6, 4:3, 5:2}` | `▅▆█▄▂` | Left-skewed (net negative) |
| `{1:4, 2:4, 3:4, 4:4, 5:4}` | `████████` | Perfectly uniform |
| `{1:0, 2:0, 3:0, 4:0, 5:20}` | `    █` | Unanimous 5 |
| `{1:20, 2:0, 3:0, 4:0, 5:0}` | `█    ` | Unanimous 1 |

### 10.4 Where Bars Appear

Distribution bars appear in:
- Summary format table (§2.1): one bar per dimension row
- Comparison table (§3.1): one bar each for A and B per row
- Detailed format output (§2.2): bars are not used (raw counts are shown instead)

The bars are a quick visual signal for skew and spread. They are generated by the Claude agent from the XML distribution data, not by the tool itself.

## 11. Discord Markdown Limitations

The following Discord markdown constraints apply to all SSR output:

### 11.1 What Works in Discord

| Formatting | Syntax | Notes |
|-----------|--------|-------|
| Bold | `**text**` | Works everywhere |
| Italic | `*text*` | Works everywhere |
| Code (inline) | `` `code` `` | Used for panel IDs, run IDs |
| Code block | ` ```code``` ` | Avoid for results — wastes space |
| Block quote | `> text` | Used for persona highlight quotes |
| Table | `\|col\|col\|\n\|---|---\|\n\|val\|val\|` | Works in desktop Discord; mobile may not render |
| Numbered list | `1. item` | Works everywhere |
| Bullet list | `• item` (or `- item`) | Works everywhere |
| Header | `# H1`, `## H2`, `### H3` | Works in desktop; smaller on mobile |

### 11.2 What Does NOT Work in Discord

| Feature | Why Not Available |
|---------|-----------------|
| Collapsible sections (`<details>`) | HTML not rendered in Discord |
| Nested tables | Not supported by Discord's table renderer |
| Links with display text `[text](url)` | Works in Discord but avoid for tool output (no external URLs in results) |
| Syntax-highlighted code blocks | Discord only supports a few languages; not useful for results |
| Images or charts | Bot cannot post rendered charts; Unicode bars are the substitute |
| Message editing | Not used in current Daimon architecture |
| Reactions for pagination | Not used in current Daimon architecture |

### 11.3 Table Formatting Notes

Discord tables require the pipe-separator format. For the dimension table in §2.1:

```
| Dimension | Mean | Mode | Distribution |
|-----------|------|------|-------------|
| Purchase Intent | 3.85/5 | 4 | ▁▂▄████▄ |
```

**Column width**: Discord auto-sizes columns. Do not add padding spaces inside cells.

**Dimension name truncation**: If a dimension name exceeds 20 characters in the table, use a short form:
| Full dimension name | Short form for table |
|--------------------|---------------------|
| Purchase Intent | Purchase Intent |
| Brand Favorability | Brand Favorability |
| Message Clarity | Message Clarity |
| Emotional Response | Emotional Resp. |
| Personal Relevance | Personal Rel. |
| Uniqueness / Differentiation | Uniqueness |
| Trust & Credibility | Trust |
| Value Perception | Value Perc. |
| Share-Worthiness | Share-Worth. |
| Overall Appeal | Overall Appeal |

### 11.4 Quote Truncation

Highlight quotes from persona responses are truncated to 200 characters in summary format. Truncation rules:
1. Cut at the nearest word boundary before 200 chars
2. Append `...` if truncated
3. Never cut mid-word or mid-sentence in the middle of a quote

In detailed format, quotes may be up to 400 chars (one message has more space per quote since it's posted in multi-message format).

## 12. Complete Error Message Pattern Summary

All SSR tool errors follow this conversational rendering pattern:
1. **Acknowledge what failed** (not "an error occurred" — name the operation)
2. **Explain in plain English** (not "ToolError: panel_id not found")
3. **Offer a next step** (retry, clarify, or alternate action)

Claude should **never** show raw error messages, exception tracebacks, or tool error codes to users. The XML `<tool-error>` element is translated into natural language before posting to Discord.

```
**Panel created (partial): Filipino Moms 30-45** (17 of 20 requested personas)
```
