## Known Limitations & Future Considerations

1. **~90+ new backend service routes required**: Every tool marked NEW that calls a non-service backend endpoint requires a new `/api/service/*` route. Four domains have ZERO existing service routes: Integrations (18 routes), Users & Team (13 routes), Analytics (1 route), Workflows (8 routes).

2. **7 existing tool bugs (all P1-P3)**: All documented in `analysis/existing-tools-audit.md` and `specs/README.md`. P1 bugs cause empty XML display tags in Slack. These should be fixed before implementing new tools.

3. **3 cross-domain security gaps**: Documented in `specs/README.md` Implementation Notes. The most critical is 6 of 7 existing service routes not validating `user_id`.

4. **2 CE-native tools**: `cheerful_improve_email_content` and `cheerful_get_thread_summary` call Claude/internal services directly — no new backend route needed.

5. **Gmail OAuth**: Connect/disconnect flows are webapp-only browser redirects. CE cannot automate these flows; it can only verify status via `cheerful_list_gmail_accounts`.

6. **SLACK_USER_MAPPING is hardcoded**: Adding new users requires updating `constants.py` in the context engine deployment. There is no dynamic user registration mechanism.

7. **`cheerful_list_campaigns` silently excludes DRAFT and COMPLETED campaigns**: The existing backend service endpoint filters status to exclude these. Documented in existing tools audit.
